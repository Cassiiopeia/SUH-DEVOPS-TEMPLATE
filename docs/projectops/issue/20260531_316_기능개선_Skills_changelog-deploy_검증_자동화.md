📝 현재 문제점
---

- `changelog-deploy` 스킬은 deploy PR 생성까지만 처리하고, PR이 실제로 automerge 됐는지 확인하는 단계가 없습니다.
- 그래서 배포할 때마다 PR 상태·automerge·deploy 반영 여부를 확인하는 일회용 Python을 `/tmp`에 즉석 생성하게 됩니다 (PR 상태 조회, 워크플로우 run 확인, deploy 브랜치 반영 확인 등).
- 매 배포마다 같은 로직을 재작성하면서 토큰이 낭비되고, 코드가 흩어져 일관성이 없습니다.
- 확인 중 `sleep` 명령이 환경에서 차단되어 대기 흐름이 자주 막힙니다.

🛠️ 해결 방안 / 제안 기능
---

- 흩어진 일회용 스크립트를 **재사용 가능한 단일 서브커맨드 하나**로 통합합니다.
- agent는 레포 정보(또는 PR 번호)만 넘기고, 반환되는 종합 JSON의 판정 결과를 보고 다음 행동을 판단합니다.
- 한 번 호출로 다음 4가지를 모두 확인합니다:
  - deploy PR의 머지 상태 (열림 / 머지됨 / 닫힘)
  - PR 본문의 릴리스 노트 존재 여부 (워크플로우가 본문을 초기화했는지)
  - 배포 자동화 워크플로우 진행 상태 (진행 중 / 성공 / 실패)
  - deploy 브랜치 반영 여부
- 결과는 상황을 한 단어로 요약한 판정값과 사람이 읽을 수 있는 요약, 그리고 다음 확인을 위한 힌트로 구성됩니다.
- 대기가 필요할 때는 `sleep` 대신 자체 페이스 조정 방식으로 재확인하도록 안내합니다.

⚙️ 작업 내용
---

- GitHub API 조회 헬퍼 3종 추가 (PR 상세 / deploy PR 자동 탐색 / 브랜치 HEAD 조회)
- 종합 상태를 판정해 JSON으로 반환하는 `deploy-status` 서브커맨드 추가
- 6가지 판정값별 후속 행동을 스킬 문서에 표로 정리
- 스킬 문서에 검증 단계 통합 및 기존 fix 흐름 교체
- 판정 로직 단위 테스트 추가
