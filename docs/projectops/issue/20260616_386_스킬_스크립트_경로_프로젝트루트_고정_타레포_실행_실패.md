🗒️ 설명
---

cassiiopeia 플러그인 스킬(github, issue, commit, report, review, troubleshoot, changelog-deploy)의 `SKILL.md`가 Python CLI 스크립트를 아래처럼 **프로젝트 루트 고정 경로**로 호출한다.

```bash
PROJECT_ROOT=$(git rev-parse --show-toplevel 2>/dev/null || pwd)
cd "$PROJECT_ROOT/skills/<skill>/scripts"   # ← 프로젝트 안에 스크립트가 있다고 가정
```

이 가정은 **SUH-DEVOPS-TEMPLATE 레포 내부에서만** 참이다. 사용자 프로젝트는 "Use this template" 초기화나 `template_integrator` 통합 시 `skills/` 폴더가 **복사 대상에서 제외**되므로, 해당 폴더가 아예 존재하지 않는다. 실제 스크립트는 플러그인 캐시에 설치된다:

```
~/.claude/plugins/cache/<marketplace>/cassiiopeia/<version>/skills/<skill>/scripts/<scope>_cli.py
```

그 결과 **다른 레포에서 스킬을 실행하면 `cd`가 실패**하고, AI가 "스크립트가 없다 → config도 없나?"로 오판하며 플러그인 캐시·config 경로를 손으로 헤매게 된다. 연쇄적으로 "config를 엉뚱한 곳에서 읽는다"는 착각까지 유발한다.

> **윈도우 특정 문제가 아니다.** 윈도우 경로(`C:\Users\...`)가 길어 눈에 띄었을 뿐, macOS·Linux에서도 동일하게 재현된다. 원인은 OS가 아니라 "스크립트가 항상 프로젝트 안에 있다"는 틀린 전제다.

**`config.py`·`github_cli.py` 자체는 정상이다.**
- `common/config.py`는 `Path.home() / ".suh-template" / "config" / "config.json"`을 정확히 가리킨다.
- `<scope>_cli.py`는 `Path(__file__).resolve().parents[3]` 기준으로 `scripts/common`을 import하므로(cwd 무관), 스크립트 파일 위치만 맞으면 `cd` 위치와 무관하게 import가 풀린다.

🔄 재현 방법
---

1. 템플릿으로 생성했거나 `template_integrator`로 통합한 일반 사용자 레포(예: `suh-project-utility`)로 이동한다.
2. github 스킬을 호출해 Secret 업데이트·이슈 조회 등 CLI 호출이 필요한 동작을 시킨다.
3. SKILL.md 지시대로 `cd "$PROJECT_ROOT/skills/github/scripts"`가 실행되지만 해당 폴더가 없어 실패한다.
4. AI가 스크립트·config 위치를 못 찾고 플러그인 캐시·홈 디렉토리를 반복 탐색하며 헤맨다.

✅ 예상 동작
---

스킬이 **어느 레포에서 실행되든** 스크립트를 찾아 정상 호출해야 한다. 스크립트 디렉토리 탐색을 "플러그인 캐시(설치 위치, 최신 버전) 우선 → 프로젝트 루트 폴백"으로 바꾼다.

```bash
SCRIPTS=$(ls -d ~/.claude/plugins/cache/*/projectops/*/skills/<skill>/scripts 2>/dev/null | sort -V | tail -1)
[ -z "$SCRIPTS" ] && SCRIPTS="$PROJECT_ROOT/skills/<skill>/scripts"
cd "$SCRIPTS" || exit 1
```

- 캐시에 설치돼 있으면(일반 사용자 레포) 캐시의 최신 버전 스크립트를 사용한다.
- 캐시에 없으면(이 템플릿 레포 자체에서 개발 중) 프로젝트 루트로 폴백한다.
- `config.py`·`<scope>_cli.py`는 수정 불필요 — 이미 위치 독립적으로 동작한다.
