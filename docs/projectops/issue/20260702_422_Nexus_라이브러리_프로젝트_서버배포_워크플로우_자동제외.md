📝 현재 문제점
---

`template_integrator.sh` / `.ps1`로 Spring 프로젝트에 템플릿을 통합하면, `spring/`의 서버 배포 워크플로우(SSH+Docker)가 **항상 함께 설치**됩니다.

- `PROJECT-SPRING-SIMPLE-CICD`
- `PROJECT-SPRING-NONSTOP-NGINX-CICD`
- `PROJECT-SPRING-NONSTOP-TRAEFIK-CICD`
- `PROJECT-SPRING-PR-PREVIEW`

그런데 **Nexus(라이브러리 publish)로 배포하는 Spring 프로젝트**는 서버에 배포하지 않고 라이브러리를 Maven 저장소에 올리는 것이 목적입니다. 이런 프로젝트에는 위 서버 배포 워크플로우가 **불필요**한데도 그대로 설치되어, 사용하지 않는 워크플로우가 레포에 남습니다.

🛠️ 해결 방안
---

Spring 서버 배포 워크플로우를 `spring/server-deploy/` 하위 폴더로 묶고, 통합 마법사가 **Nexus 라이브러리 프로젝트(`--nexus`)일 때 이 폴더를 통째로 제외**하도록 개선합니다.

- **일반 Spring 프로젝트**: 서버 배포 워크플로우 기본 포함 (기존과 동일)
- **Nexus 라이브러리 프로젝트**: 서버 배포 워크플로우 자동 제외, 라이브러리 publish(GitHub Packages / nexus)만 유지

**확장성**: 폴더 기반 분류이므로, 앞으로 새로운 서버 배포 워크플로우를 추가할 때 `spring/server-deploy/`에 파일만 넣으면 마법사가 자동으로 동일 규칙을 적용합니다. 마법사 코드를 다시 수정할 필요가 없습니다.

빌드 검증·라이브러리 publish 성격의 워크플로우(`GITHUB-PACKAGES-PUBLISH`, `nexus/`)는 서버 배포가 아니므로 이 폴더에 넣지 않고 기존 위치에 유지합니다.

📁 영향 범위
---

- `template_integrator.sh` — server-deploy 하위폴더 처리 로직 추가
- `template_integrator.ps1` — 동일 로직 추가 (Windows)
- `.github/workflows/project-types/spring/` — 서버 배포 4개를 `server-deploy/`로 이동
- `CLAUDE.md`, `docs/SSH-DOCKER-DEPLOYMENT-GUIDE.md` — 폴더 구조·규칙 문서화
