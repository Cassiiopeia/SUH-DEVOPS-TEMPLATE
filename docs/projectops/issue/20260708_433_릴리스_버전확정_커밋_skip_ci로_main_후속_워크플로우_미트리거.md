🗒️ 설명
---

릴리스(develop→main PR automerge)가 완료돼도 **main push로 트리거돼야 할 배포·동기화 워크플로우가 전부 실행되지 않는 버그**가 있습니다.

- 영향 워크플로우: `PROJECT-TEMPLATE-NPM-PUBLISH`, `PROJECT-COMMON-README-VERSION-UPDATE`, `PROJECT-TEMPLATE-PLUGIN-VERSION-SYNC`, 그리고 각 프로젝트의 배포 CICD 전부
- 원인: `PROJECT-COMMON-AUTO-CHANGELOG-CONTROL`이 만드는 **버전 확정 커밋**(`v{버전} 릴리즈 버전 확정 ... [skip ci]`)이 develop→main fast-forward 머지로 **main의 HEAD 커밋**이 됨. GitHub Actions는 push HEAD 커밋 메시지의 `[skip ci]`를 보고 **그 push로 트리거될 모든 워크플로우를 스킵**한다.
- 결과: 릴리스마다 npm 배포·README 버전 갱신·플러그인 매니페스트 동기화·서버 배포가 자동으로 안 되고, 매번 수동 `workflow_dispatch`가 필요했다.

🔄 재현 방법
---

1. develop에 커밋 후 develop→main 릴리스 PR을 automerge (`/projectops:changelog-deploy`)
2. AUTO-CHANGELOG가 `v{버전} ... [skip ci]` 버전 확정 커밋을 만들고 main에 머지
3. main push가 발생하지만 `[skip ci]`로 인해 NPM-PUBLISH·README-UPDATE·PLUGIN-SYNC·CICD가 하나도 트리거되지 않음

📸 참고 자료
---

- 실측: v3.0.190 확정 커밋 `81c9729`로 트리거된 push 이벤트 run **0개** (`GET /actions/runs?head_sha=81c9729` 결과 workflow_dispatch 1건만 존재 — 수동 실행분)
- develop을 트리거로 쓰는 워크플로우는 저장소에 **하나도 없음** → `[skip ci]` 제거 시 develop push에서 중복 CI가 돌 우려 없음

✅ 예상 동작
---

- 릴리스 버전 확정 커밋이 main에 도달하면, `[skip ci]` 없이 push되어 배포·동기화 워크플로우가 **자동 트리거**되어야 한다 (default 브랜치 push = 배포 트리거).
- VERSION-CONTROL 안전망은 `paths-ignore(version.yml)` + `release_guard(커밋에 version.yml 변경 포함 여부 감지)` 이중 가드로 이 커밋을 릴리스로 인식해 **재bump하지 않아야** 한다 (무한 루프 없음).
- README-UPDATE·PLUGIN-SYNC는 자기가 만드는 커밋에 `[skip ci]`를 유지하므로 서로 재트리거하지 않아야 한다.
