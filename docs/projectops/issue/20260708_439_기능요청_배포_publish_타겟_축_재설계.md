📝 현재 문제점
---

**하나의 프로젝트 타입 안에 배포 방식이 여러 갈래로 갈리는데, 템플릿이 이를 표현할 구조가 없다.** 그리고 지금의 임시 해법(Spring `--nexus`)은 배포 방식을 타입에 하드코딩해 개념적으로 맞지 않는다.

실제 사례 (사용 중인 passQL 레포):
- `PROJECT-COMMON-VERCEL-DEPLOY.yml` — **Vercel 배포**를 쓰고 있으나 템플릿이 전혀 다루지 않음.
- 즉 react 타입은 Docker+SSH로도, Vercel로도 배포 가능한데 템플릿은 Docker만 안다.

핵심 문제 두 가지:

1. **타입 ≠ 배포 방식.** 같은 타입이 여러 배포 타겟을 가질 수 있다.
   - react → `docker` | `vercel` (상호배타, 택1)
   - node → `server`(docker) | `npm-publish` (공존 가능)
   - spring → `server` | `nexus` | `github-packages`

2. **배포 타겟을 타입에 종속시키면 또 틀린다.** `nexus`는 spring만의 것이 아니고(다른 JVM/비-JVM도 가능), `npm`은 node만의 것이 아니다(react 라이브러리도 npm publish). 배포방식을 타입 키에 매핑하는 순간 다시 부정확해진다.

3. **마법사가 "이게 무슨 성격의 프로젝트인지" 판단할 수 없다.** 현재 감지는 "package.json 있음" 같은 **파일 존재**로만 타입을 정하지, "서버냐 / 라이브러리냐 / vercel 앱이냐"라는 **배포 의도**를 알 방법이 없다. 이건 타입으로도, 하드코딩 축으로도 풀리지 않는다.

관련 자산:
- `.github/workflows/project-types/spring/nexus/` + `--nexus` (기존 유일한 배포타겟 분기)
- passQL의 `PROJECT-COMMON-VERCEL-DEPLOY.yml` (템플릿화 대상 후보)
- `template_integrator.sh`의 `per_type_workflow_copy` (`server-deploy/` 기본포함 + `nexus/` opt-in 처리)

🛠️ 해결 방안 / 제안 기능
---

**배포/publish 타겟을 타입과 독립된 축으로 재설계한다. (방향 미확정 — 설계 필요)**

이 이슈는 **설계 이슈**다. 구현 전에 다음을 확정해야 한다.

검토 중인 방향 (아직 결정 안 됨):
- "서버 배포 방식"(택1)과 "라이브러리 publish"(opt-in)를 개념적으로 분리할지
- 배포 타겟을 어떻게 타입-비종속으로 표현할지 (마법사가 사용자에게 물어보는 방식이 유력 — 파일로 자동 판단 불가하므로)
- `version.yml` 스키마: `deploy_target` / `publish_targets` 등 새 키를 어떻게 둘지
- 기존 Spring `--nexus`를 새 축으로 흡수(마이그레이션)할지 여부와 기존 프로젝트 호환 검증
- Vercel 워크플로우를 템플릿 자산으로 편입 (react 배포 타겟으로)

**주의: 이 축을 성급히 설계하면 또 틀린다** (실제로 검토 중 타입-종속 설계가 반복적으로 부정확했음). 충분히 브레인스토밍 후 별도 스펙을 작성한다.
