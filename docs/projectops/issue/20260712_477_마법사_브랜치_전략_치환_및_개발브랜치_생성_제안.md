# ⚙️[기능추가][통합마법사] 브랜치 전략 실효화 — 워크플로우 브랜치 치환 및 개발 브랜치 생성 제안

> 라벨: 작업전
> 담당자: Cassiiopeia

🗒️ 설명
---

- 마법사는 이미 기본 브랜치(자동 감지+수정 메뉴)와 배포 브랜치(`deploy_branch`, 릴리스 PR head — 기본 develop)를 질문·저장(#456·#457)하지만, **워크플로우는 그 값을 쓰지 않습니다**:
  - 트리거가 `branches: ["main"]` / `branches: [develop]`로 하드코딩 — 기본 브랜치가 main이 아니거나 개발 브랜치명이 develop이 아니면 CI/CICD/릴리스가 전부 무반응
  - `PROJECT-COMMON-RELEASE-CHANGELOG`의 automerge 가드가 `if: head.ref == 'develop'` 하드코딩 — `deploy_branch`를 다르게 설정한 레포의 릴리스 PR이 **조용히 스킵**됨
  - 내부 step에도 `git fetch origin main` 등 리터럴 잔존
- 또한 마법사가 완료 요약에서 "develop 브랜치를 만드세요"라고 안내만 할 뿐, **브랜치 존재 확인·생성을 해주지 않습니다** — 사용자가 이 단계를 빠뜨리면 릴리스 파이프라인이 동작하지 않습니다.

🛠️ 해결 방안
---

1. **브랜치 치환 모듈** (`src/core/branch-sub.js`): 브랜치 설정이 표준(main/develop)과 다를 때만, 복사된 워크플로우의 트리거(`branches:` 인라인/멀티라인)·head 가드·`git fetch origin main`을 설정값으로 교체. 표준값이면 no-op(바이트 동일 유지).
2. **복사 엔진 배선**: `copyWorkflows` 마지막에 복사된 파일 전체(post-pass) 치환 + `isUnchanged` 가상 비교에 브랜치 치환 포함(업데이트 시 재복사 churn 방지).
3. **내부 리터럴 동적화**: RELEASE-CHANGELOG의 step 내부 `origin main`·안내 문구를 `github.event.pull_request.base.ref` 기반으로 교체 (트리거 외에는 치환 불필요하게 — 템플릿 자체도 이득).
4. **개발 브랜치 생성 제안** (대화형): `deploy_branch` 확정 후 로컬/원격 존재 확인 → 없으면 "기본 브랜치에서 만들까요?" → 로컬 생성, push 여부 별도 확인(실패 시 명령어 안내). 비대화형은 안내만.
5. 완료 요약의 "develop 브랜치 생성" 고정 문구를 실제 설정 브랜치명·생성 여부 기반으로 갱신.

✅ 기대 동작
---

- 기본 브랜치 master + 개발 브랜치 release인 레포에 통합하면: CI는 release push에, CICD·릴리스는 master에 반응하고, release→master 릴리스 PR이 automerge 파이프라인을 정상 통과
- 표준(main/develop) 레포는 파일 바이트 변화 없음 (기존 동작 100% 보존)
- 개발 브랜치가 없는 레포는 마법사가 물어보고 만들어줌
