> 라벨: 작업전
> 담당자: Cassiiopeia

🗒️ 설명
---

Spring 프로젝트 통합 대화형 흐름(실측: suh-project-utility, v4.2.12)에서 질문 문구·안내가 사용자를 혼란시키는 지점 4건. 모두 동작 버그가 아니라 문구·안내·초기값 표현 문제이며 `src/core/options-ask.js`에 몰려 있다.

### 1. publish 선택에 "라이브러리 배포 안 함" 명시 항목이 없다

- 현재: multiselect에서 아무것도 토글하지 않고 Enter = "없음". 유도 문구도 "없으면 그냥 Enter".
- 문제: "그냥 Enter"는 사용자가 무엇을 스킵했는지 확신하기 어렵다. 명시적으로 "안 함"을 고르는 감각이 없다.
- 개선: `◻ 라이브러리 배포 안 함` 같은 명시 항목을 넣거나, 선택 없이 Enter 시 "라이브러리 배포 안 함으로 진행합니다"를 분명히 표기. "그냥 Enter" 표현 제거.

### 2. CodeRabbit "사용" 선택 후 후속 설정 안내가 없다 (가장 실질적)

- 현재: "CodeRabbit AI 코드 리뷰 사용 → 예" 고르면 `CodeRabbit 코드 리뷰: 사용`만 찍고 끝.
- 문제: CodeRabbit은 토글만으로 동작하지 않는다. coderabbit.ai 접속 → GitHub 앱 설치 → 해당 레포에 **grant access** 를 해야 실제 PR 리뷰가 붙는다. 안내가 없으면 사용자가 "켰으니 되겠지"로 넘어가 실제론 동작하지 않는다.
- 개선: "사용" 선택 시 "https://coderabbit.ai 에서 GitHub 앱을 설치하고 이 저장소에 접근 권한(grant access)을 부여해야 실제로 리뷰가 붙습니다" 안내를 그 자리에서 출력. (완료 요약에도 재노출)

### 3. changelog 생성기 질문이 "하나만 골라야" 하는 것처럼 보인다 — 실제로는 폴백 사다리

- 현재: single select로 github-ai / coderabbit / openai / commit 중 하나 선택.
- 문제: 실제 `ladder.py`는 "선택 provider → github-ai → commit" 순으로 **자동 폴백**한다(#455). 그런데 질문 UX는 폴백이 있다는 사실을 전혀 드러내지 않아, 사용자가 "하나 잘못 고르면 실패하나?"라고 오해한다.
- 개선: 유도 문구에 "고른 방식이 실패하면 자동으로 GitHub AI → 커밋 분석 순으로 폴백하니 안심하고 고르세요"를 명시. 선택은 "1순위 생성기를 고르는 것"임을 표현.

### 4. 배포 브랜치 질문이 develop 하드코딩 + 내부자 문맥 — 동적 감지 미반영

- 현재:
  ```
  🌿 릴리스 배포 브랜치(릴리스 PR의 head)는 무엇인가요?
     develop→main 릴리스 구조면 develop 그대로 두세요. 배포 브랜치가 따로면 그 이름을 적어주세요.
  ◆ 배포 브랜치  [초기값: develop 하드코딩]
  ```
- 문제:
  - "develop→main 릴리스 구조면 develop 그대로 두세요"는 처음 통합하는 사용자가 알아들을 수 없는 내부자 설명이다.
  - 초기값이 감지와 무관하게 항상 `develop`이다. 오늘 방향(#477)은 "기본 브랜치를 git으로 감지해 동적으로 안내"였는데 이 질문이 그걸 반영하지 못한다.
- 개선:
  - git으로 감지한 기본 브랜치를 함께 보여준다. 예: `감지된 기본 브랜치: main · 릴리스는 보통 그 앞단(개발) 브랜치에서 올립니다`
  - 선택지/초기값을 `develop (추천)` 형태로 제시하되, 감지된 기본 브랜치가 표준과 다르면 그에 맞춰 안내를 조정.
  - "릴리스 PR의 head" 같은 용어 대신 평이한 설명("개발한 걸 모아서 배포로 올리는 브랜치")으로.

🛠️ 해결 방안
---

- 위 4건 모두 `src/core/options-ask.js`의 질문 유도 문구·선택지·초기값 표현을 개선한다. 저장 로직·폴백 사다리·브랜치 치환 동작은 변경하지 않는다.
- 2번(CodeRabbit 후속 안내)과 4번(브랜치 동적 감지)은 사용자 실동작에 직접 영향을 주므로 우선순위가 높다.
- 4번은 `ensureDeployBranch`(#477)와 연계 — 감지된 기본 브랜치를 질문 단계에서부터 노출한다.
