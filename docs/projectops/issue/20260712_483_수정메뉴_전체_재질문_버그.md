# ❗[버그][통합마법사] 수정 메뉴에서 "배포/Publish 방식"만 골라도 옵션 6종이 전부 재질문됨

> 라벨: 작업전
> 담당자: Cassiiopeia

🗒️ 설명
---

분석 개요 카드에서 "수정하기" → "배포/Publish 방식"만 선택했는데, 배포 방식 하나만 다시 묻는 게 아니라 **배포 → publish → CodeRabbit → changelog → 배포 브랜치(생성 제안 포함) → (그리고 Secret)** 까지 옵션 질문이 전부 다시 나온다. 사용자는 "한 항목만 고치려 했는데 왜 처음부터 다 다시 입력하냐"며 혼란을 겪는다 (실측: suh-project-utility, v4.2.12).

### 원인 (코드 확인)

- `src/ui/prompts.js:42` — 수정 메뉴 항목은 `deploy-publish`("배포/Publish 방식") **하나**뿐이다.
- `src/commands/interactive.js:162` — 이 항목을 고르면 `askAllOptionalWorkflows(..., forceAsk: true)`를 호출한다.
- `src/core/options-ask.js` — 이 함수는 deploy·publish·codeReviewCoderabbit·changelogProvider·changelogBaseUrl·**deployBranch**·secretBackup을 **한 덩어리로** 묻는다. `forceAsk: true`면 저장값이 있어도 **모든 질문을 다시** 던진다.

즉 "배포/Publish 방식"이라는 **한 메뉴 = 옵션 6~7종 전체 재질문**으로 매핑되어 있다. 수정 항목과 실제 재질문 범위가 1:N으로 어긋난다.

특히 배포 브랜치 재질문은 `ensureDeployBranch`(#477)까지 다시 타서, "배포/Publish만 고치려 했는데 develop 브랜치 생성 제안이 또 뜨는" 이상 동작을 낳는다.

📸 참고 자료 (현재 화면)
---

```
◇  어떤 항목을 수정할까요?
│  배포/Publish 방식        ← 이것만 골랐는데

🚀 이 프로젝트를 어디에 배포하나요?      ← deploy 다시
📦 라이브러리로 배포(publish)할 계획?    ← publish 다시
◆  배포 브랜치 develop                   ← 브랜치 다시
⚠️ 배포 브랜치 'develop'가 없습니다...   ← 생성 제안까지 다시
```

🛠️ 해결 방안
---

1. **수정 메뉴를 실제 항목 단위로 분해**한다. 예:
   - "배포 방식(deploy)"
   - "라이브러리 배포(publish)"
   - "CodeRabbit 코드 리뷰"
   - "changelog 생성기"
   - "릴리스 소스(개발) 브랜치"  ← #482 네이밍 정리와 연계
   - "Secret 백업"
   메뉴가 많아지면 "배포/자동화 옵션" 하위 그룹으로 2단 메뉴 구성도 가능.
2. **또는** `askAllOptionalWorkflows`에 "어떤 축만 재질문할지" 스코프 인자를 추가해, 수정 메뉴 선택에 해당하는 질문만 `forceAsk`되도록 한다(나머지는 저장값 유지).
3. 어느 쪽이든 **"한 항목 수정 = 그 항목만 재질문"** 이 되도록 한다. 통짜 재질문 제거.

✅ 기대 동작
---

- 수정 메뉴에서 고른 항목만 다시 묻고, 나머지 옵션은 이전에 정한 값을 유지한다.
- 배포 방식만 바꾸려는데 브랜치 생성 제안이 다시 뜨지 않는다.
