# ⚙️[기능추가][통합마법사] 의도 우선(intent-first) 진입 구조 재설계 — 프로젝트 성격을 먼저 묻고 옵션 유도

> 라벨: 작업전
> 담당자: Cassiiopeia

🗒️ 설명
---

#480~483에서 마법사의 배포/publish·브랜치·수정 메뉴 질문을 문구·순서·격리 관점에서 다듬었지만, 그 과정에서 더 근본적인 구조 문제가 드러났다:

> **마법사가 "이 프로젝트가 뭘 하는 것인지"를 안 묻고, "이 옵션 쓸래? 저 옵션 쓸래?"를 낱개로 순차 질문한다.**

그래서 사용자가 매 질문마다 "이게 나한테 해당되나?"를 스스로 판단해야 한다. 실측(suh-project-utility, Spring 서버 앱): 서버 앱 개발자한테 "라이브러리(Nexus/npm) 배포할래?"를 묻고, deploy 선택지에 "라이브러리/CI 전용" 같은 다른 축 용어가 섞여 혼란을 준다. #480의 "두 축 안내 문구 추가"는 증상 완화이지 구조 변경이 아니다.

🛠️ 해결 방안
---

**intent(프로젝트 성격)를 맨 앞에 한 번 묻고, 답에 따라 관련 옵션만 좁혀서 물어본다.**

- 진입 질문: 서버 앱 / 라이브러리 / 둘 다 / 배포 안 함 / 직접 하나씩(기존 방식).
- intent → 축 유도:
  - `app` → deploy만 물음, publish=[] 자동
  - `library` → publish만 물음, deploy=none 자동
  - `both` → 둘 다
  - `none` → 둘 다 none/[]
  - `manual` → 현행 순차 질문 전부 (탈출구·하위호환)
- code-review·changelog·릴리스 브랜치·secret은 intent와 무관하게 유지(팀 워크플로우 선택이므로).
- basic 단독 타입은 지금처럼 intent 질문도 스킵.

**하위호환**:
- version.yml `options.intent` 신설(추가만), 기존 deploy/publish 키는 병기.
- 구 version.yml(intent 없음)은 deploy/publish 값에서 intent 역추론.
- 비대화형 `--intent` 플래그 추가, 미지정 시 `--deploy`/`--publish`에서 역추론 → 기존 CLI 무수정 동작.
- 수정 메뉴(#483)에 "프로젝트 성격" 항목 추가, 개별 축 항목은 유지.

✅ 기대 동작
---

- 사용자는 "내 프로젝트가 뭔지"만 답하면 관련 옵션만 좁혀 물어본다.
- 무관한 옵션은 조용히 기본값 확정.
- `manual`로 언제든 기존 방식 그대로 가능.

⏳ 예상 소요 시간
---

- 6~8시간 (진입 흐름·저장·역추론·수정 메뉴·CLI·테스트).
