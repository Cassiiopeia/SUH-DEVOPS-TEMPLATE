🗒️ 설명
---

npx 마법사(대화형 통합)에서 프로젝트 타입이 잘못 감지되었을 때 사용자가 빠져나갈 방법이 없고, 나중에 타입을 수정해도 이미 설치된 구 타입 워크플로우가 정리되지 않는 버그입니다. 실사용(suh-project-control 레포, v4.2.14)에서 실측된 문제입니다.

**문제 1: 경로 확정 단계에 타입 탈출구가 없음 (강제 수락 함정)**

- 아카이브 폴더(code-archive/) 안의 build.gradle이 maxdepth 3 스캔에 걸려 spring으로 오감지되었습니다.
- 경로 확정 단계(resolveProjectPaths)에서 후보 경로를 거부하면 직접 입력 루프로 들어가는데, 이 루프에는 "이 타입이 아니다, 제외하겠다"는 선택지가 없습니다.
- 유효한 경로를 입력하거나, "그래도 이 경로를 사용할까요?"에 예를 눌러 잘못된 경로를 강제 수락할 때까지 무한 반복됩니다.
- 실제로 사용자가 두 번 거부하다가 결국 예를 눌러 spring이 루트(.) 경로로 잘못 확정되었고, spring 배포 워크플로우 4개가 설치되었습니다.

**문제 2: 타입 변경 후 구 타입 워크플로우가 고아로 잔존**

- 재실행 후 수정 메뉴에서 타입을 spring,python에서 python으로 바꿔도, 첫 실행 때 설치된 PROJECT-SPRING-*.yaml 4개(SIMPLE-CICD, NONSTOP-NGINX, NONSTOP-TRAEFIK, PR-PREVIEW)는 그대로 남습니다.
- 복사 엔진은 선택된 타입만 순회하며 복사할 뿐, 선택 해제된 타입의 기존 워크플로우를 감지하거나 정리하는 로직이 없습니다.
- 결과적으로 python 단독 프로젝트에 spring 배포 파이프라인이 살아있는 상태가 됩니다.

🔄 재현 방법
---

1. 아카이브성 하위 폴더에 build.gradle이 있는 비 spring 프로젝트에서 npx projectops 실행
2. spring이 자동 감지됨, 경로 확정 단계에서 제안된 spring 루트를 "아니오"로 거부
3. 직접 입력 루프 진입, 어떤 경로를 넣어도 마커 파일이 없다는 경고와 재입력만 반복 (타입 제외 선택지 없음)
4. "그래도 사용할까요?"에 예를 눌러야만 진행 가능, spring 워크플로우가 설치됨
5. 마법사 재실행, 수정 메뉴에서 타입을 python 단독으로 변경
6. 완료 후 .github/workflows/ 확인: PROJECT-SPRING-*.yaml이 그대로 잔존

✅ 예상 동작
---

- 경로 확정 단계의 후보 확인, 후보 선택, 직접 입력 실패 지점 모두에서 "이 타입이 아님 (제외)"을 선택할 수 있어야 합니다. 제외하면 해당 타입이 project_types, 워크플로우 복사, env 질문에서 모두 빠지고, 마지막 타입까지 제외되면 basic으로 폴백해야 합니다.
- 타입, 경로 확정 후 선택되지 않은 타입의 템플릿 워크플로우가 대상 레포에 실재하면 고아로 감지하여 레거시 정리와 동일한 UX(목록 표시, 확인 후 .bak 무해화, 복원 가능)로 정리를 제안해야 합니다. 파일명은 템플릿 인벤토리와 정확히 일치하는 것만 대상으로 하여 사용자 커스텀 워크플로우를 건드리지 않아야 합니다. 비대화형(--force)에서는 자동 무해화 없이 안내만 출력합니다.
