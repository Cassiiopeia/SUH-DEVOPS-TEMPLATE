📝 현재 문제점
---

- 마법사(특히 **업데이트 통합**)는 레포의 워크플로우·설정을 크게 바꾸지만, **무엇이 바뀌었고 사람이 무엇을 확인해야 하는지가 터미널 출력으로만 흘러가고 휘발**됩니다.
- 레거시 레포 마이그레이션은 결국 사람이 수동으로 마무리해야 하는 항목이 많습니다:
  - 구세대 배포 워크플로우(SYNOLOGY 등)가 "현역일 수 있어" 자동 삭제되지 않고 남음 → 신형 전환 후 직접 삭제 필요
  - `.bak` 무해화된 레거시 파일 확인 후 정리
  - 새로 설치된 CICD가 요구하는 GitHub Secrets 등록
  - env 값(JDK, 도메인, 포트 등)이 실제 환경과 맞는지 검증
  - 기존 커스텀 워크플로우와 신형의 트리거 중복/충돌 여부 확인
- 나중에 **AI Agent가 이 레포를 열었을 때** "마법사가 언제 어떤 버전에서 어떤 버전으로 뭘 바꿨는지"를 알 수 있는 구조화된 기록이 없습니다. `version.yml`의 `metadata.template`은 최종 옵션만 담고, 실행 이력·변경 파일 목록·보류된 수동 작업은 담지 않습니다. 그래서 AI가 "마법사 업데이트로 뭐가 바뀌었으니 이 부분 충돌 안 나게 가다듬어야겠다"는 판단을 할 근거가 없습니다.
- **특히 v4는 v1~v3 대비 바뀐 것이 매우 많습니다** (브랜치 전략 전면 전환, Skills 마켓플레이스 전환, npx 단일 CLI 전환, project_type SSOT, deploy/publish 축 재설계 등). 기존 1·2·3버전대를 쓰던 많은 레포가 4버전으로 점프하면 **CRITICAL breaking change 여러 개를 한꺼번에 통과**하는데, 이 내용이 실행 중 터미널에서 "상세 내용 볼까요? → 아니오"로 스킵되면 그대로 사라집니다. 마이그레이션 후 사용자가 AI Agent를 돌려 "뭘 확인해야 하는지" 물어봐도 근거 문서가 없습니다.

🛠️ 해결 방안 / 제안 기능
---

마법사 실행(full/workflows 모드)이 끝나면 대상 레포에 **`PROJECTOPS-MIGRATION-GUIDE.md`** (레포 루트, SETUP-GUIDE 옆)를 자동 생성/append 한다. 사람용 가이드와 AI Agent용 메타데이터를 한 파일에 담는다.

**1) 파일 구조 — 고정 헤더 + 실행 엔트리 append**

- **고정 헤더** (최초 1회): 문서 목적, 사람용 읽는 법, **AI Agent 해석 가이드라인**
  - AI 가이드라인 예: "아래 엔트리의 yaml 블록을 읽어라. `leftover_old_gen`이 있으면 신형 워크플로우와 트리거 중복 여부를 검사하고 전환/삭제를 제안하라. `skipped_conflict`가 있으면 사용자 커스텀본과 신형 템플릿의 차이를 비교해 병합을 제안하라. `env_applied` 값이 실제 워크플로우 env와 다르면 드리프트를 경고하라. `manual_actions_pending`이 비어 있지 않으면 사용자에게 남은 수동 작업을 상기시켜라."
- **실행 엔트리** (매 실행 append, `---` 구분자): 사람용 섹션 + AI용 메타데이터 블록

**2) 엔트리의 사람용 섹션 — 실행 결과 기반 동적 체크리스트**

해당 없는 항목은 아예 출력하지 않는다 (실행에서 실제 발생한 것만):
- 템플릿 버전 from→to, 모드, 타입, 변경 요약 (신규 N / 교체(.bak) N / 스킵 N)
- ☐ 구세대 배포 워크플로우 전환 후 삭제 (파일명·신형 대응 명시)
- ☐ `.bak` 무해화 파일 확인 후 정리
- ☐ 새 CICD가 요구하는 Secrets 등록 (워크플로우별 필요 secret 목록)
- ☐ env 적용값 검증 (JDK/도메인/포트 등 — 적용된 실제 값 표시)
- ☐ 4.3.0 폐기 예고 등 deprecated 항목
- ☐ 개발 브랜치 생성 여부 (마법사가 만들었으면 "완료됨" 표시)

**3) 엔트리의 AI용 메타데이터 — 스키마 버저닝된 yaml 펜스 블록**

````
```yaml
# projectops-migration (machine-readable)
schema: 1
run_at: "2026-07-14T00:16:04Z"
template: { from: "2.7.7", to: "4.2.15" }
mode: full
types: [spring]
options: { deploy: docker-ssh, publish: [], secret_backup: true, intent: app }
branches: { default: main, deploy: develop, deploy_branch_created: true }
workflows:
  added: [...]
  replaced_bak: [...]          # .bak 백업 후 교체
  skipped_conflict: [...]      # 사용자 수정본 유지 — 병합 검토 후보
  legacy_neutralized:          # 구형 → .bak 무해화
    - { file: "...", replacement: "...", reason: "..." }
  leftover_old_gen:            # 현역 가능성으로 보존 — 사람이 전환 후 삭제
    - { file: "...", replacement: "..." }
env_applied: { spring: { JAVA_VERSION: "17", SERVICE_DOMAIN: "..." } }
breaking_acknowledged: ["2.9.0", "3.0.186", "4.1.0", "4.0.0"]
manual_actions_pending: [delete-old-gen-workflows, register-secrets]
```
````

- `schema: 1` 버저닝으로 미래 필드 추가에 안전
- 데이터 소스는 이미 존재하는 실행 결과들: `copyWorkflows` counters(copiedFiles/skipped/bak), 레거시 정리 결과, 고아 정리 결과, `envValues`, `ensureDeployBranch` ready, breaking check 목록 — 각 단계 결과를 하나의 runReport 객체로 모아 렌더링

**3.5) 버전 점프(메이저 마이그레이션) 지식 임베드 — v1~v3 → v4 사용자 핵심 대응**

터미널에서 "상세 내용 스킵"해도 가이드 문서에는 **통과한 breaking change 전체의 조치 방법이 반드시 기록**된다:

- 통과한 breaking change를 `.github/config/breaking-changes.json`에서 severity·title·**message(조치 방법) 전문**과 함께 엔트리에 임베드 — 예: 2.7.7→4.2.15 점프 시 2.9.0(Skills 마켓플레이스 전환), 3.0.133(SSH 인증 방식), 3.0.137(SYNOLOGY 폐기), 3.0.186(브랜치 전략 전환 — deploy 브랜치 폐기), 4.0.0(npx CLI 전환), 4.1.0(project_type SSOT), 4.2.0(deploy/publish 축) 7건 전부
- 각 breaking change에 **"이 레포에서의 실제 영향"을 실행 결과와 연결** — 예: "3.0.186 브랜치 전환 → 이 레포는 develop이 없어 마법사가 생성함 / 기존 deploy 브랜치 기반 워크플로우 X.yaml이 잔존함", "3.0.137 SYNOLOGY 폐기 → 이 레포에 구세대 파일 2개 잔존 (leftover_old_gen 참조)"
- AI용 메타 블록에도 대응 필드 추가:
  ```yaml
  breaking_traversed:
    - { version: "3.0.186", severity: critical, title: "브랜치 전략 전면 전환", affected_in_this_repo: ["deploy 브랜치 참조 워크플로우 잔존"], action_required: true }
  ```
- 효과: 마이그레이션 직후 사용자가 AI Agent에게 "이 레포 마이그레이션 뭐 확인해야 해?"라고 물으면, Agent가 이 문서 하나로 **버전 점프에서 통과한 변경 전체 + 이 레포에 실제로 남은 수동 작업**을 근거 있게 안내·수행할 수 있다

**4) 배치·수명 규칙**

- 대상 레포 자산이므로 커밋 대상 (gitignore 제외 안 함) — 팀원·AI가 공유
- 템플릿 전용 파일 아님: initializer 삭제 목록·integrator 복사 제외 목록에 넣지 않는다 (마법사가 대상 레포에 만들어 주는 파일)
- append 전용: 기존 엔트리를 절대 수정하지 않는다 (이력 보존). 파일이 없으면 헤더부터 생성
- 완료 요약에 "📖 마이그레이션 가이드: PROJECTOPS-MIGRATION-GUIDE.md" 한 줄 안내 추가
