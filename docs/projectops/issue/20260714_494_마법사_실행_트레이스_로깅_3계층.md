📝 현재 문제점
---

- 마법사 실행 과정(질문·답변·파일별 복사/스킵/.bak 결정·env 치환·브랜치 생성)이 **터미널 출력으로만 흘러가고 어디에도 남지 않습니다.**
- 문제가 생기면 사용자가 터미널 로그를 수동으로 복사해 AI Agent에게 붙여줘야 진단이 가능합니다 (실측: v4.2.15 마이그레이션 진단이 정확히 이 방식이었고, #489~#491 버그 발견의 근거가 사용자가 복사해 준 로그였음).
- 원시 로그는 AI가 매번 파싱해야 하고, UI 문구가 바뀌면 해석이 깨집니다. "왜 이 파일이 .bak 됐지?", "이 env 값이 왜 이렇지?" 같은 질문에 파일 단위 인과를 추적할 구조화된 데이터가 없습니다.
- #493(마이그레이션 가이드)은 큐레이션된 요약이라, 상세 디버깅·인과 추적용 데이터까지 담으면 체크리스트가 로그에 파묻혀 문서가 오염됩니다.

🛠️ 해결 방안 / 제안 기능
---

마법사 실행 기록을 **3계층으로 분리**해 남긴다. 가이드는 "읽는 문서", 로그는 "찾아보는 데이터".

```
docs/projectops/migration/
├── PROJECTOPS-MIGRATION-GUIDE.md            ← Layer 1: 큐레이션 요약 (#493)
├── 20260714_001604_v2.7.7_to_v4.2.15.jsonl  ← Layer 2: 구조화 트레이스 (이 이슈의 핵심)
└── 20260714_001604_v2.7.7_to_v4.2.15.log    ← Layer 3: 터미널 원문 미러
```

**Layer 2 — JSONL 이벤트 트레이스 (핵심)**

마법사의 모든 결정·행동을 이벤트 한 줄씩 기록한다. 스키마 버저닝(`schema: 1`) 적용:

```jsonl
{"ts":"2026-07-14T00:15:31Z","phase":"detect","action":"type-detected","target":"spring","detail":{"marker":"build.gradle"}}
{"ts":"...","phase":"ask","action":"answered","target":"deploy","detail":{"question":"실행물 배포 방식","answer":"docker-ssh","source":"user"}}
{"ts":"...","phase":"copy","action":"copied","target":"PROJECT-SPRING-SIMPLE-CICD.yaml","detail":{"kind":"new"}}
{"ts":"...","phase":"copy","action":"skipped","target":"PROJECT-COMMON-CI.yaml","detail":{"reason":"unchanged"}}
{"ts":"...","phase":"legacy","action":"neutralized","target":"PROJECT-COMMON-AUTO-CHANGELOG-CONTROL.yaml","detail":{"to":".bak","replacement":"PROJECT-COMMON-RELEASE-CHANGELOG.yaml"}}
{"ts":"...","phase":"env","action":"substituted","target":"PROJECT-SPRING-SIMPLE-CICD.yaml","detail":{"key":"VOLUME_HOST_PATH","before":"/volume1/projects/__PROJECT_NAME__","after":"/volume1/projects/suh-project-utility"}}
{"ts":"...","phase":"branch","action":"created-pushed","target":"develop","detail":{"from":"main"}}
{"ts":"...","phase":"warn","action":"leftover-old-gen","target":"PROJECT-SPRING-SYNOLOGY-PR-PREVIEW.yaml","detail":{"replacement":"PROJECT-SPRING-PR-PREVIEW.yaml"}}
```

- 파일명 grep 한 번으로 "이 파일에 마법사가 한 모든 일"이 시간순으로 나온다 — 파일 단위 인과 추적
- env `substituted` 이벤트의 before/after 기록으로 #489류(치환 불일치) 버그가 즉시 드러난다
- run당 수 KB — 커밋 대상이라 fresh clone에서도 AI가 전체 이력 조회 가능

**Layer 3 — 터미널 원문 미러 (.log)**

- 사용자가 본 출력 그대로 저장 (오늘 수동 복사한 그 로그의 자동화)
- 마법사 출력이 이미 `say`/`log` 함수를 경유하므로 tee 방식으로 미러링 — 크래시 시에도 부분 기록이 남아 포렌식 가능

**Layer 1 연결 (#493과의 관계)**

- 가이드 엔트리의 yaml 메타 블록에 `trace_file`/`log_file` 경로 포인터 추가
- AI 활용 라우팅: "뭐 확인해야 해?" → Layer 1 / "왜 이 파일이 이래?" → Layer 2 grep / "버그 재현 상황 보여줘" → Layer 3

**수명·안전 규칙**

- run당 파일 1쌍 (타임스탬프+버전 파일명) — 기존 기록 불변(append-only 이력)
- 민감값 기록 금지: PAT·secret 값은 이벤트에 절대 넣지 않는다 (env 치환값은 비민감 — version.yml deploy 블록과 동일 기준)
- 사용자 프로젝트 자산이므로 initializer/integrator 제외 목록에 넣지 않는다
