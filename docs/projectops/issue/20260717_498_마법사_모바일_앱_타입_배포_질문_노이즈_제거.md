📝 현재 문제점
---

- flutter 단독 프로젝트에 `npx projectops` 통합 시 마법사가 "실행물(서버/앱)을 어디에 올리나요?" (docker-ssh/vercel/none) 질문을 그대로 노출한다 (실측: elum 레포 통합 로그).
- flutter에는 `server-deploy/` 폴더 자체가 없어 `docker-ssh`를 골라도 `none`을 골라도 **복사되는 파일이 하나도 달라지지 않는다** — 순수 노이즈 질문이다. `vercel`을 고르면 모바일 앱에 무의미한 Vercel 워크플로우가 들어가는 해악만 있다.
- 질문이 오답을 유도한다: Flutter "앱"이니까 성격(intent) 질문에서 "앱·서비스"를 고르게 되고, 그러면 deploy 질문이 나와 기본값 `docker-ssh`가 version.yml에 박힌다. Flutter의 실제 배포(Play Store/TestFlight/Firebase)는 이 축과 무관하게 타입 워크플로우로 항상 포함되는데도 마치 서버 배포 설정이 필요한 것처럼 보인다.
- publish 축(nexus/npm/github-packages)도 flutter에 해당하는 타겟이 하나도 없다. react-native/react-native-expo도 동일한 문제다.
- 근본 원인: intent 우선 분기(#485)가 "프로젝트 타입" 정보를 전혀 쓰지 않는다. `basic` 단독만 특례로 스킵할 뿐(`isBasicOnly`), 사실상 같은 처지인 모바일 앱 타입들은 특례가 없다.

🛠️ 해결 방안 / 제안 기능
---

- **타입별 적용 가능 deploy/publish 타겟 선언** 도입 (`src/core/options-ask.js`):
  - `TYPE_DEPLOY_TARGETS` / `TYPE_PUBLISH_TARGETS` 선언 맵 신설. 빈 배열 = 그 축이 개념상 성립하지 않는 타입.
  - 서버형 타입(spring/react/node/python)은 현행 선택지 그대로 (동작 무변경).
  - 모바일 앱 타입(flutter/react-native/react-native-expo)과 basic은 두 축 모두 빈 배열.
- 선택된 타입 집합의 **합집합**으로 적용 가능 타겟을 산출:
  - 두 축 모두 비어 있으면(모바일/basic 단독 등) intent·deploy·publish 질문을 전부 스킵하고 `none`/`[]`로 조용히 확정한다. 안내 문구도 띄우지 않는다 (스토어 배포는 타입 워크플로우가 이미 커버하므로 사용자에게 보여줄 이유 자체가 없음).
  - flutter+spring 같은 멀티타입은 spring 때문에 축이 살아 있으므로 질문 유지.
  - 기존 `isBasicOnly` 특례는 이 일반화된 판정으로 흡수.
- **저장값 조용한 정리**: version.yml에 이미 적용 불가 값이 저장돼 있으면(예: flutter 단독 + `deploy: docker-ssh`) 경고 없이 `none`/교집합으로 정리한다. 복사 결과가 동일하므로 안전하며, SSOT 원칙(무의미 값 잔존 금지)에 부합.
- 비대화형(`npx projectops --force` 등, `src/index.js`)도 동일하게 적용 불가 값을 정리.
- 수정 메뉴(`src/ui/prompts.js` editMenu): 적용 불가 축의 항목("프로젝트 성격", "배포 방식", "라이브러리 배포 타겟")을 숨긴다. 타입을 바꾸면 다음 메뉴부터 다시 나타난다.
- 질문 선택지도 적용 가능 타겟으로 필터링 (현재 선언으로는 서버형 타입 무변경이지만, 이후 타입별 선택지 세분화의 기반).
