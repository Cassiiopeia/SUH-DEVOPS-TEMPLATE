# projectops 리브랜딩 및 npm 배포 계획

## 1. 한 줄 요약
현재 리포지토리(`suh-github-template`)의 명칭 및 에이전트 스킬들의 개인 맞춤형 명칭(SUH)을 범용적인 `projectops`로 리브랜딩하고, npx로 즉시 실행 가능한 CLI 마법사 패키지 형태로 npm에 배포/선점한다.

## 2. 배경
"SUH-DEVOPS-TEMPLATE" 및 "SUH-" 접두어는 개인용 템플릿이라는 인상을 줍니다. 누구나 쉽게 가져다 쓰고 오픈소스로서 확장성을 갖추기 위해, "프로젝트 운영 자동화"를 상징하는 프로페셔널한 명칭인 `projectops`로 리브랜딩이 필요합니다. 또한, 사용자들이 여러 단계의 스크립트 실행 대신 `npx projectops` 한 줄로 자신들의 프로젝트에 이 강력한 자동화 환경을 이식할 수 있도록 마법사를 npm에 성공적으로 등재해야 합니다.

## 3. 사용자 시나리오 / 동작 정의
- **시나리오 1 (이름 선점)**: 관리자가 `projectops`라는 고유 명칭으로 최소한의 CLI 스텁 패키지를 npm 레지스트리에 배포하여 이름을 선점한다.
- **시나리오 2 (마법사 실행)**: 개발자가 새로운 또는 기존 프로젝트의 터미널에서 `npx projectops`를 입력한다.
- **시나리오 3 (인터랙티브 설치)**: 마법사가 터미널에 뜨고 "어떤 DevOps 요소(GitHub Actions, 이슈 템플릿, AI 에이전트 스킬 등)를 추가하시겠습니까?"라고 질문한다. 선택을 완료하면 관련 설정 파일이 개발자의 프로젝트 루트에 깨끗하게 설치된다.

## 4. 요구사항
**필수 (Must)**:
- npm 레지스트리에 `projectops` 명칭으로 배포가 가능하도록 `package.json` 및 CLI 엔트리포인트를 구성한다.
- `npx projectops` 실행 시 환영 메시지와 기본적인 마법사 UI 진입이 가능한 실행 파일을 준비한다.
- 리포지토리 내의 핵심 문서(`README.md`, 스크립트 등)에 있는 "SUH" 관련 텍스트와 파일명을 `projectops`에 맞춰 전면 개편한다.
- npm 배포 시 불필요한 설정 파일이나 내부용 폴더가 올라가지 않도록 이중 필터링을 설정한다.

**원함 (Should)**:
- 마법사 내에서 인터랙티브한 UI(예: 질문 및 선택지 선택)를 제공할 수 있도록 대화형 패키지(Prompts 등)의 의존성을 고려한다.
- 기존의 배시(`template_integrator.sh`) 및 파워쉘(`template_integrator.ps1`) 설치 스크립트의 유효성을 유지하면서 CLI와의 시너지 방안을 수립한다.

**선택 (Nice)**:
- 추후 릴리스를 완전 자동화할 수 있는 GitHub Actions 워크플로우를 보강한다.

## 5. 제약
- **기술**: Node.js ESM (ES Modules) 형식을 채택하여 최신 패키지 규격을 준수하고, CLI 실행을 위한 Shebang(`#!/usr/bin/env node`) 규칙을 반드시 지킨다.
- **환경**: 크로스 플랫폼(Windows, macOS, Linux)에서 마법사가 오작동 없이 작동해야 하므로, OS 종속적인 쉘 명령어 사용을 금지하고 Node.js 내장 API 또는 크로스 플랫폼 라이브러리만을 활용한다.
- **인증**: npm 배포를 위한 2FA(이중 인증) 절차 및 토큰 관리가 필요하다.

## 6. 성공 기준 (Definition of Done)
- [ ] `package.json` 내 패키지명이 `projectops`로 변경되고, `bin` 필드가 정상적으로 연결되어 로컬 테스트를 통과한다.
- [ ] `npx projectops` 실행 시 CLI 도구의 안내 및 환영 프롬프트가 정상 작동한다.
- [ ] 전체 리포지토리의 리브랜딩(SUH -> projectops) 영향 범위가 문서상 정의된다.
- [ ] npm에 최초 `projectops@0.0.1` 버전을 성공적으로 publish하여 이름을 확보한다.
