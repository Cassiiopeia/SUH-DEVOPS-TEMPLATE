# 테스트 빌드 버전 0.0.0 하드코딩으로 인한 TestFlight 업로드 실패

## 개요

앱스토어 정식 출시 이후 Apple이 `0.0.0` prerelease train을 영구 폐쇄하면서, 테스트 빌드 워크플로우(iOS TestFlight, Android APK)의 업로드가 실패하는 문제 수정. 하드코딩된 `0.0.0` 버전을 `version_manager.sh`를 통한 동적 버전 읽기로 변경하여, 프로덕션 워크플로우와 동일한 버전 관리 방식을 적용함. RomRom-FE 프로젝트(TEAM-ROMROM/RomRom-FE#735)에서 동일 문제 발생 후 검증 완료된 패턴을 템플릿에 반영.

## 변경 사항

### iOS 테스트 빌드 (PROJECT-FLUTTER-IOS-TEST-TESTFLIGHT.yaml)
- `버전 관리 스크립트 권한 설정` step 추가: `chmod +x ./.github/scripts/version_manager.sh`
- `테스트 빌드 버전 설정` step: `echo "version=0.0.0"` → `version_manager.sh get | tail -n 1`로 동적 읽기
- `Flutter build` step: `--build-name="0.0.0"` → `--build-name="${{ needs.prepare-test-build.outputs.version }}"`
- 진행상황 댓글 7곳: `0.0.0(${buildNumber})` → `${version}(${buildNumber})`로 변경
- notify-start/prepare-실패 댓글: 버전을 아직 모르는 시점이므로 `빌드 번호`만 표시하도록 변경
- 콘솔 알림: `echo "버전: 0.0.0"` → 동적 버전 출력

### Android 테스트 빌드 (PROJECT-FLUTTER-ANDROID-TEST-APK.yaml)
- job outputs에 `version` 추가: `${{ steps.build_info.outputs.version }}`
- `버전 관리 스크립트 권한 설정` step 추가: `chmod +x ./.github/scripts/version_manager.sh`
- `빌드 정보 생성` step: `version_manager.sh get | tail -n 1`로 버전 추출 후 output 전달
- `Set test build version` step: `0.0.0+${BUILD_NUMBER}` → `${VERSION}+${BUILD_NUMBER}`로 pubspec.yaml 수정
- 진행상황 댓글 4곳(성공/실패/이슈 댓글): `0.0.0(${buildNumber})` → `${version}(${buildNumber})`

### 빌드 트리거 (PROJECT-FLUTTER-SUH-LAB-APP-BUILD-TRIGGER.yaml)
- 주석 수정: `테스트 버전 0.0.0 고정으로 운영 버전과 분리` → `version.yml에서 동적으로 읽어 사용`

## 주요 구현 내용

### 버전 읽기 방식 통일

프로덕션 워크플로우(PLAYSTORE-CICD, TESTFLIGHT, FIREBASE-CICD)와 동일한 `version_manager.sh get` 방식을 채택. RomRom-FE에서는 `grep + sed` 직접 파싱을 사용했으나, 템플릿에서는 version.yml ↔ pubspec.yaml 동기화/검증 기능이 내장된 `version_manager.sh`를 활용하여 더 안정적인 버전 관리를 제공.

```bash
# 프로덕션/테스트 워크플로우 공통 패턴
VERSION=$(./.github/scripts/version_manager.sh get | tail -n 1)
```

### 빌드번호 정책 유지

버전명만 변경하고 빌드번호는 기존 PR/이슈 번호 기반 고유 번호 정책 유지. Apple/Google 정책상 같은 버전명 내에서 빌드번호가 겹치지 않으면 업로드 가능.

- 테스트 빌드: `1.9.101(73500)` (이슈/PR 번호 기반 빌드번호)
- 운영 빌드: `1.9.101(245)` (version_code 기반 빌드번호)

### notify-start 시점 버전 미표시

notify-start job은 ubuntu-latest에서 checkout 없이 실행되므로 `version_manager.sh`를 실행할 수 없음. 따라서 초기 댓글에는 빌드번호만 표시하고, prepare-test-build 완료 후 업데이트되는 댓글부터 실제 버전 표시.

## 주의사항

- 앱스토어에 새 버전이 출시될 때마다 해당 버전 이하의 prerelease train이 닫히는 것은 Apple 정책. 현재 수정으로 version.yml의 최신 버전을 따라가므로 운영 배포 직후에도 자동으로 더 높은 버전으로 빌드됨
- Android 테스트 빌드는 Play Store에 올라가지 않아 원래 문제가 없었으나, iOS와 일관성을 위해 동일하게 수정
- `version_manager.sh get`은 version.yml과 프로젝트 파일(pubspec.yaml 등)의 버전을 비교하여 높은 버전을 반환하므로, 버전 불일치 상황에서도 안전하게 동작
