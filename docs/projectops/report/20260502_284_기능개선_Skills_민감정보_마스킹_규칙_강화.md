# 🚀[기능개선][Skills] 민감정보 마스킹 규칙 강화 — DB유저명 누락 및 *** 익명별표 허용 문제

## 개요

`skills/references/common-rules.md`의 민감정보 마스킹 규칙에 DB 유저명 항목 누락, `***` 익명 별표 허용 문구, 일부 skill의 자체검토 프로토콜 누락 등 세 가지 문제가 있었다. 실제 운영 DB 계정명이 문서에 그대로 노출되는 사고를 계기로 규칙을 보완하고, 파일을 저장하는 모든 관련 skill에 자체검토 참조를 추가했다.

## 변경 사항

### 마스킹 규칙 강화 (common-rules.md)

- `skills/references/common-rules.md`: 마스킹 규칙 표에 `DB 유저명 / 계정명 → {DB_USERNAME}, {ADMIN_ID}` 항목 추가
- `skills/references/common-rules.md`: 자체검토 항목 표에 `username: kimchi` 형태 계정명 노출을 위험 예시로 추가
- `skills/references/common-rules.md`: "스크린샷/로그 인용 시 `***` 또는 플레이스홀더로 대체" 문구 → `***` 익명 별표 금지, 명명된 플레이스홀더만 허용으로 교체
- `skills/references/common-rules.md`: 판단 기준에 `{DB_USERNAME}` 안전 예시, 계정명 노출 위험 예시, `********` 금지 명시 추가

### 자체검토 프로토콜 누락 보완

- `skills/plan/SKILL.md`: 산출물 저장 섹션에 파일 저장 직전 자체검토 프로토콜 참조 추가
- `skills/analyze/SKILL.md`: 동일
- `skills/ppt/SKILL.md`: 동일

## 주요 구현 내용

### 문제 발생 원인

`common-rules.md` 마스킹 규칙 표에 비밀번호·API Key·서버 주소는 있었으나 **DB 유저명(계정명)** 항목이 없었다. AI가 `username: {DB_USERNAME}` 형태를 위험으로 판단하지 못해 실제 계정명이 보고서에 그대로 기재됐다.

또한 "스크린샷/로그 인용 시 `***` 또는 플레이스홀더로 대체"라는 문구가 있어 AI가 명명된 플레이스홀더 대신 `********` 같은 익명 별표를 사용하는 경우가 발생했다. `********`는 어떤 종류의 정보인지 알 수 없어 문서 재현성이 떨어진다.

### 자체검토 범위 확대

기존에는 `issue`, `report` skill만 "파일 저장 직전 자체검토 프로토콜" 참조가 명시되어 있었다. 파일을 저장하는 나머지 skill(`plan`, `analyze`, `ppt`)에도 동일한 참조 문구를 추가해 모든 산출물 파일에서 민감 정보 검토가 수행되도록 범위를 확대했다.

### 마스킹 기준 명확화

| 항목 | 변경 전 | 변경 후 |
|------|---------|---------|
| DB 유저명 | 규칙 없음 | `{DB_USERNAME}` 플레이스홀더 사용 |
| 익명 별표 | `***` 허용 | 금지 — 명명된 플레이스홀더만 허용 |
| 계정명 예시 | 없음 | `username: kimchi` → 위험 예시 추가 |

## 주의사항

- 이 변경은 skill 규칙 파일만 수정했다. 기존에 이미 저장된 문서(`RomRom-BE/docs/suh-template/` 등)에 노출된 실제 계정명은 별도로 수동 교체가 필요하다.
- `plan`, `analyze`, `ppt` 외에도 파일을 저장하는 skill이 추가되는 경우 동일하게 자체검토 참조 문구를 포함해야 한다.
