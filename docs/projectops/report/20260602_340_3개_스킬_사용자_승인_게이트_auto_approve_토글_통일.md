## 무엇을 / 왜 바꿨는가

`commit` / `issue` / `changelog-deploy` 3개 스킬은 작업 직전 사용자 확인을 받는 게이트가 있는데, `changelog-deploy`만 자동 모드 토글(`changelog_deploy.auto_approve_release_notes`)이 있었고 키 이름·구조가 스킬별로 달라 일관성이 없었다. 같은 레포에서 반복 작업할 때 매번 동일한 확인이 떠 마찰이 컸다.

3 스킬을 **단일 패턴**으로 통일했다. py는 무변경, 모든 변경은 SKILL.md + 표준 config 스키마에서 흡수.

## 변경 사항

### 1) Config 스키마 통일

```jsonc
{
  "github": {
    "global_pat": "...",
    "commit":            { "auto_approve": false },
    "issue":             { "auto_approve": false },
    "changelog_deploy":  { "auto_approve": false },
    "repos": [
      {
        "owner": "...", "repo": "...",
        "commit":            { "auto_approve": true },
        "issue":             { "auto_approve": true },
        "changelog_deploy":  { "auto_approve": true }
      }
    ]
  }
}
```

해석 우선순위 (3 스킬 공통):
```
1. repos[i].{skill_id}.auto_approve   (현 owner/repo 매칭)
2. github.{skill_id}.auto_approve     (글로벌 기본값)
3. false                              (안전 default — 수동 승인)
```

### 2) SKILL.md A/B/C 분기 표준화

- **A. 자동 모드**: 요약/본문 표시 후 즉시 다음 단계. 사용자 응답 대기 없음
- **B. 수동 모드**: 기존 게이트(스킬별 옵션 차이) 그대로
- **C. 첫 실행 자동화 제안**: `CONFIG_HAS_KEY == false`일 때 B 분기 승인 직후 1회만 "이 레포 / 모든 레포 / 매번 확인" 선택. agent가 Read/Write로 config 직접 갱신

### 3) UX 원칙

- 사용자 메시지에 `auto_approve`·`config.json`·키 이름 노출 금지
- 자연어 토글만:
  - "자동으로 진행해줘" / "확인 없이 해줘" → `true`
  - "매번 확인받게 해줘" / "수동으로 바꿔줘" → `false`
- agent가 Read/Write로 config 갱신, 사용자는 손 안 댐

### 4) issue 특이사항

자동 모드라도 **중복 검사(2-1, 4-1단계) 강제**. open 동일 이슈 발견 시 무조건 중단. 자동 모드는 "최종 등록 승인"만 스킵.

### 5) 마이그레이션 — 명시적 break

구 키 `changelog_deploy.auto_approve_release_notes`는 더 이상 인식하지 않는다. config에 남아 있어도 무시. 자동 모드를 원하면 1회 토글 발화로 `auto_approve: true` 새로 저장.

## py 변경
