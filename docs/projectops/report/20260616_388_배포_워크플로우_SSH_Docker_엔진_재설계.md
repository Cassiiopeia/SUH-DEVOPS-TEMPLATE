# 배포 워크플로우를 확장 가능한 SSH+Docker 엔진으로 재설계

## 개요

기존 배포 워크플로우는 `synology/` 폴더에 격리되고 파일명·변수·옵션·설정 전반에 `SYNOLOGY` 식별자가 박혀 있어, "이건 Synology NAS 전용"이라는 잘못된 신호를 주고 있었다. 그러나 실측 결과 이 워크플로우들은 모두 **"SSH 접속 → Docker 이미지 pull → 컨테이너 교체"**라는 단일 패턴이며, Synology냐 AWS EC2냐의 실제 차이는 ① SSH 인증 방식(비밀번호 vs `.pem` 키), ② sudo 처리(`echo $PW | sudo -S` vs passwordless `sudo`), ③ 경로 기본값 세 가지뿐이다. 즉 사람이 서버를 쓴다면 그게 Synology든 AWS EC2든 GCP든 일반 VPS든 **"그냥 SSH로 접속하는 서버"**일 뿐, Synology를 별도 카테고리로 구분할 이유가 없었다.

이 작업은 두 축으로 진행되었다. 첫째, **SSH 인증 방식을 확장**해 `SSH_AUTH_METHOD`(password/key) 환경변수와 `SUDO()` 헬퍼 분기를 도입, 비밀번호 인증 서버(Synology 등)와 키 인증 서버(AWS EC2 등)를 워크플로우 복제 없이 같은 엔진으로 처리한다. 둘째, **`synology` 식별자를 프로젝트 전반에서 전면 제거**한다 — 폴더·파일명·주석·env변수·통합 마법사 UX/옵션·`version.yml` 스키마에서 synology를 걷어내고, 배포 워크플로우는 각 타입 루트로 올려 기본 포함하며, 성격이 다른 Nexus(라이브러리 publish)·Secret 백업은 독립 게이트(`--nexus`/`--secret-backup`)로 분리했다. 결과적으로 새 배포처가 추가돼도 워크플로우를 복제할 필요 없이 `SSH_AUTH_METHOD`와 경로만 지정하면 되는 **확장 가능한 SSH+Docker 배포 엔진**이 되었다.

## 기능 흐름

배포 step의 `appleboy/ssh-action`에는 `password`와 `key`를 **둘 다 전달**하고, 액션이 채워진 쪽만 사용한다. 원격 서버에서는 `SSH_AUTH_METHOD` 값에 따라 `SUDO()` 헬퍼를 password/key 모드로 정의해 이후 모든 docker 명령을 추상화한다.

```mermaid
flowchart TD
    A[배포 트리거 main/deploy push] --> B[appleboy/ssh-action]
    B --> C["password·key secret 둘 다 전달<br/>envs: SSH_AUTH_METHOD"]
    C --> D{"SSH_AUTH_METHOD<br/>(기본값 password)"}

    D -->|password| E["SERVER_PASSWORD secret로 접속<br/>(Synology·일반 서버)"]
    D -->|key| F["SSH_KEY(.pem) secret로 접속<br/>(AWS EC2 등)"]

    E --> G["SUDO() { echo \$PW | sudo -S \"\$@\"; }<br/>sudo 비밀번호 stdin 주입"]
    F --> H["SUDO() { sudo \"\$@\"; }<br/>passwordless sudo 직접 실행"]

    G --> I[SUDO docker pull]
    H --> I
    I --> J[SUDO docker rm -f 기존 컨테이너]
    J --> K[SUDO docker run -d 신규 컨테이너]
    K --> L[SUDO docker ps / logs 헬스체크]
    L --> M{정상 기동?}
    M -->|성공| N[배포 완료]
    M -->|실패| O["SUDO docker logs --tail 50<br/>exit 1"]
```

핵심은 인증·sudo 차이를 `SUDO()` **단일 추상화 지점**으로 흡수했다는 것이다. 빌드(gradle→docker→push), pull→컨테이너 교체, 포트/볼륨/헬스체크 로직은 인증 방식과 무관하게 **글자 단위로 동일**하게 유지된다.

## 변경 사항

### SSH 인증 방식 확장

- 배포 워크플로우 4종(SPRING SIMPLE / NONSTOP-NGINX / NONSTOP-TRAEFIK / PR-PREVIEW)에 `SSH_AUTH_METHOD` env(`@wizard ask`, 기본값 `password`)와 `SSH_KEY` secret 안내를 추가했다. (`0dc619c`, `c0731ad`, `21b9a8d`, `60d7b8e`)
- 배포 step에서 `appleboy/ssh-action`에 `password`·`key`를 동시 전달하고 `envs: SSH_AUTH_METHOD`로 값을 원격 셸까지 넘긴다. 액션이 빈 값을 무시하므로 인증 방식에 맞는 쪽만 사용된다.
- 원격 스크립트 도입부에 `SSH_AUTH_METHOD` 기반 `SUDO()` 헬퍼 분기를 두고, 기존의 `echo $PW | sudo -S docker ...` / `sudo docker ...` 호출을 모두 `SUDO docker ...`로 치환했다. password 모드는 sudo 비밀번호를 stdin으로 주입하고, key 모드는 passwordless sudo를 직접 호출한다.
- `template_integrator`(.sh/.ps1)에 `SSH_AUTH_METHOD` 기본값 `password`를 등록해 마법사 통합 시에도 동일 기본값이 적용되도록 했다. (`117460e`, `d254549`)

### synology 식별자 제거 / 워크플로우 재구성

- 워크플로우 파일명·설정 내부의 `SYNOLOGY` 식별자를 일괄 제거하고 SIMPLE-CICD 명명을 표준화했다. (`c97ae06`)

### integrator(sh/ps1) 옵션 게이트 전환

- `--synology` **단일 게이트를 제거**하고 `--nexus`/`--no-nexus`, `--secret-backup`/`--no-secret-backup`(`.ps1`은 `-Nexus`/`-NoNexus`, `-SecretBackup`/`-NoSecretBackup`) 두 독립 게이트로 분리했다. 표준 SSH+Docker 배포 워크플로우는 질문 없이 **기본 포함**한다. (`c7c6392`)
- 변수명을 `INCLUDE_SYNOLOGY` → `INCLUDE_NEXUS`/`INCLUDE_SECRET_BACKUP`(sh), `$script:IncludeSynology` → `$script:IncludeNexus`/`$script:IncludeSecretBackup`(ps1)로 분리했다.
- synology 전용 `ask_synology_option()`을 **폴더 경로·아이콘·라벨·설명·include 변수명을 인자로 받는 범용 `ask_optional_workflow()`**로 교체하고, 이를 nexus·secret-backup 두 게이트에 순차 적용하는 `ask_all_optional_workflows()`로 묶었다. 향후 보조 워크플로우가 추가돼도 같은 코드로 처리된다. (`f9a81a7`)
- 복사·확인 메뉴·요약 출력 로직을 nexus/secret-backup으로 전환하고 synology 식별자를 완전히 제거했다. 폴더 스캔 경로를 `_td/nexus`(Nexus 게이트)·`common/secret-backup`(Secret 백업 게이트)로 갱신했다. (`f7af5a7`)
- `version.yml`의 `metadata.template.options` 키를 `synology` → `nexus`/`secret_backup`으로 전환했다. **하위호환 없이 새 키만** 읽고 쓰며, 구 `synology` 키는 어느 분기에서도 매핑하지 않는다. (`3600295`)
- `.ps1`을 `.sh`와 대칭으로 동일하게 재구성했다(215줄 추가/210줄 삭제). `-Nexus`/`-SecretBackup` 파라미터, `$script:Include*` 변수, 확인 메뉴 라벨·요약 출력("📦 Nexus publish : 포함/제외", "🔐 Secret 백업 : 포함/제외")을 sh와 일치시켰다. (`7cec1c1`)

### 문서·breaking-changes

- `breaking-changes.json`에 **3.0.137** 항목(severity `warning`)을 등록했다. synology 폴더/옵션 제거, 배포 워크플로우 루트화·기본 포함, Flutter SELFHOSTED 개명, Nexus/Secret 백업 분리, `version.yml` 옵션 키 변경, AWS EC2 키 인증 설정법을 안내한다. SSH 인증 방식 추가 안내도 함께 등록했다. (`d4e9280`, `886f64c`)
- `docs/SYNOLOGY-DEPLOYMENT-GUIDE.md`를 `docs/SSH-DOCKER-DEPLOYMENT-GUIDE.md`로 개명·범용화하고, SSH 인증 방식과 AWS EC2 배포 가이드를 추가했다. README/CLAUDE.md 및 관련 문서의 synology 표현·링크·파일명을 정리했다. (`8af7c6c`, `6a19acf`)

## 주요 구현 내용

### 1. SSH_AUTH_METHOD env + SUDO() 헬퍼 패턴

워크플로우 env에 인증 방식을 선언하고, secret 안내에 password/key 두 경우를 명시한다.

```yaml
# SERVER_PASSWORD: SSH 비밀번호 (SSH_AUTH_METHOD=password일 때)
# SSH_KEY: SSH 개인키 .pem 내용 (SSH_AUTH_METHOD=key일 때, AWS EC2 등)
env:
  # 🔐 SSH 인증 방식: "password"(Synology·일반 서버) | "key"(AWS EC2·.pem)
  SSH_AUTH_METHOD: "password"  # @wizard ask: SSH 인증 방식(password/key) [기본: password]
```

배포 step은 password·key를 동시 전달하고 `SSH_AUTH_METHOD`를 원격 셸로 넘긴다.

```yaml
- name: Deploy
  uses: appleboy/ssh-action@v1.0.3
  env:
    SSH_AUTH_METHOD: ${{ env.SSH_AUTH_METHOD }}
  with:
    host: ${{ secrets.SERVER_HOST }}
    username: ${{ secrets.SERVER_USER }}
    password: ${{ secrets.SERVER_PASSWORD }}   # password 모드용 (빈 값이면 무시)
    key: ${{ secrets.SSH_KEY }}                # key 모드용 (빈 값이면 무시)
    port: ${{ env.SSH_PORT }}
    envs: SSH_AUTH_METHOD
    script: |
      set -e
      export PW=${{ secrets.SERVER_PASSWORD }}

      # 🔐 SSH 인증 방식에 따른 sudo 추상화
      SSH_AUTH_METHOD="${SSH_AUTH_METHOD:-password}"
      if [ "${SSH_AUTH_METHOD}" = "key" ]; then
        SUDO() { sudo "$@"; }
      else
        SUDO() { echo "$PW" | sudo -S "$@"; }
      fi
```

이후 모든 docker 호출이 `SUDO docker pull`, `SUDO docker rm -f`, `SUDO docker run -d`, `SUDO docker ps`, `SUDO docker logs ...` 형태로 통일되어, 인증 방식 차이가 한 곳에서만 결정된다.

### 2. 범용 ask_optional_workflow() 게이트

synology 전용 질문 함수를 폴더·라벨·변수명을 인자로 받는 범용 함수로 일반화했다(bash 3.2 호환을 위해 nameref 대신 `eval` 사용).

```bash
# 인자: [--force-ask] $1=폴더경로 $2=아이콘 $3=짧은이름 $4=한줄설명 $5=include변수명
ask_optional_workflow() {
    local _force_ask=false
    if [ "$1" = "--force-ask" ]; then _force_ask=true; shift; fi
    local _dir="$1" _icon="$2" _short="$3" _desc="$4" _varname="$5"
    local _cur; eval "_cur=\"\${$_varname}\""
    [ -d "$_dir" ] || return
    # 폴더 내 파일 0개면 조용히 return, 이미 설정값 있으면(비-force) 건너뜀, TTY 없으면 기본 제외
    ...
    if ask_yes_no "$_short 워크플로우를 포함할까요?" "N"; then
        eval "$_varname=true"
    else
        eval "$_varname=false"
    fi
}

# Nexus(각 타입 nexus/) + Secret 백업(common/secret-backup/)을 순차 질문
ask_all_optional_workflows() {
    ...
    for _td in "${type_dirs[@]}"; do
        ask_optional_workflow $_fa "$_td/nexus" "📦" "Nexus 라이브러리 publish" \
            "라이브러리/모듈을 Maven 저장소(Nexus)에 배포하는 워크플로우입니다." INCLUDE_NEXUS
    done
    ask_optional_workflow $_fa "$_common_root/secret-backup" "🔐" "Secret 서버 백업" \
        "GitHub Secret에 저장한 설정 파일을 SSH로 서버에 업로드·이력관리하는 워크플로우입니다." \
        INCLUDE_SECRET_BACKUP
}
```

### 3. version.yml 새 옵션 키 (하위호환 없음)

읽기·쓰기 모두 `nexus`/`secret_backup` 두 키만 처리한다. 한 키만 읽고 끝내지 않도록 `continue`로 둘 다 스캔하며, 구 `synology` 키는 어느 분기에도 매핑되지 않는다.

```yaml
metadata:
  template:
    options:
      nexus: true|false           # Nexus publish 워크플로우 포함 여부
      secret_backup: true|false   # Secret 백업 워크플로우 포함 여부
```

`.ps1`도 동일하게 `-Nexus`/`-SecretBackup` 파라미터, `$script:IncludeNexus`/`$script:IncludeSecretBackup` 변수, 확인 메뉴·요약 출력을 sh와 대칭으로 구현했다.

## 주의사항

- **하위호환 없음 — 새 키만 사용**: `version.yml`의 옵션 키가 `options.synology`에서 `options.nexus` / `options.secret_backup`으로 바뀌었으며, 구 `synology` 키를 자동 매핑하지 않는다(읽기/쓰기 모두 새 키만 처리). 기존에 synology 옵션을 설정해 둔 프로젝트는 **통합 마법사(`template_integrator`)를 다시 실행해 새 옵션(Nexus / Secret 백업 포함 여부)을 명시적으로 다시 설정**해야 한다.
- **breaking-changes 등록됨(3.0.137, severity `warning`)**: 폴더 구조 변경·옵션 키 변경·파일 개명 내역이 등록되어 사용자에게 안내된다. 표준 배포 워크플로우는 루트로 이동해 자동 포함되므로 배포 동작 자체는 무중단이라 `critical`이 아닌 `warning`으로 분류했다.
- **기존 synology 옵션 사용자 영향**: `--synology`/`--no-synology` 옵션과 `-IncludeSynology`류 플래그가 제거되었다. 이를 스크립트/CI에서 호출하던 경우 `--nexus`·`--secret-backup`(`.ps1`은 `-Nexus`·`-SecretBackup`)으로 교체해야 한다.
- **배포 회귀 없음**: `SSH_AUTH_METHOD` 기본값이 `password`이고 `SUDO()` password 모드가 기존 `echo $PW | sudo -S`와 동일하게 동작하므로, 기존 Synology/비밀번호 인증 서버 배포는 100% 그대로 유지된다. AWS EC2 등 키 인증 서버는 `SSH_AUTH_METHOD=key` env + `SSH_KEY` secret(.pem 전체 내용)을 설정하면 된다.
- **파일 개명 참조 갱신 필요**: 배포 가이드가 `docs/SYNOLOGY-DEPLOYMENT-GUIDE.md` → `docs/SSH-DOCKER-DEPLOYMENT-GUIDE.md`로 개명되었으므로, 외부에서 이 경로를 링크하던 문서는 갱신이 필요하다. Flutter 자체서버 APK 배포는 `PROJECT-FLUTTER-ANDROID-SELFHOSTED-CICD`로 개명되었다.
- **워크플로우 실행 로직 무손상**: 주석·`name:`·문서 텍스트만 중립화했고 `run:`/`uses:`/`with:`/`steps:` 실행 로직은 변경하지 않았다. `synology` 식별자는 별개 스킬명(`synology-expose`)·`ssh` 예시 등 이번 범위 밖 항목에는 그대로 두었다.
