# config 탐색 방어 표기 추가로 PAT 오인 재질문 방지

## 개요

config.json은 `{HOME}/.suh-template/config/config.json` 한 곳으로 고정된 경로인데, 같은 SKILL.md 안에서 스크립트(`*_cli.py`) 위치를 찾을 때 쓰는 `ls ~/.claude/plugins/cache/...` 캐시 탐색 패턴을 에이전트가 config 찾기에까지 잘못 전이시켜, 고정 경로를 바로 Read하지 않고 플러그인 캐시 안을 뒤지는 사고방식이 발생했다. 그 결과 이미 등록된 PAT가 있는데도 "config 없음"으로 오판해 사용자에게 PAT를 다시 묻는 일이 반복됐다. 이번 변경은 단일 진실 원천인 `config-rules.md §3`의 "config 탐색 금지" 규칙을 눈에 띄는 경고 블록으로 격상하고, config를 다루는 7개 스킬 문서에 짧은 인라인 방어 표기를 추가해 참조 문서를 읽지 않고 시작해도 캐시 탐색 실수가 막히도록 이중 방어한 것이다. 스크립트 탐색용 `ls` 실행 로직 자체는 건드리지 않고, 설명/안내 텍스트만 추가했다.

## 변경 사항

### config-rules.md 경고 블록 격상

- `skills/references/config-rules.md` §3(Config 읽기) 상단에 `⚠️` 경고 블록을 신규 추가해 단일 진실 원천을 강화했다.
- 핵심 문구:
  - config.json은 `{HOME}/.suh-template/config/config.json` **한 곳으로 고정**이며, Read tool로 그 경로를 **바로** 읽는다 — `ls`·`find`·glob으로 찾지 않는다.
  - 특히 플러그인 캐시(`~/.claude/plugins/cache/...`)를 뒤지지 않는다. 각 스킬의 `ls ~/.claude/plugins/cache/*/projectops/*/skills/.../scripts` 패턴은 **스크립트(`*_cli.py`) 전용**이며 config.json은 그 캐시 안에 **없다**.
  - 그 `ls` 패턴을 config 찾기에 전이시키면 엉뚱한 파일을 잡거나 "config 없음"으로 오판해 이미 등록된 PAT를 다시 묻게 된다(실제 발생한 사고).
  - 원칙 한 줄로 정리: **"스크립트는 캐시에서 ls로, config는 홈의 고정 경로에서 Read로 — 두 경로를 절대 섞지 않는다."**

### 7개 스킬 SKILL.md 인라인 방어 추가

config 또는 PAT를 다루는 진입 지점에 짧은 인라인 방어 표기를 추가했다. 참조 문서를 읽지 않고 바로 작업을 시작해도 캐시 탐색 실수가 막히도록 하는 이중 방어다.

- `skills/issue/SKILL.md`: Config 확인 단계(§2~5)에 `⚠️` 블록 추가 — 고정 경로 Read, `ls`·`find`·glob 금지, `issue_cli.py` 탐색용 캐시 `ls` 패턴은 스크립트 전용이고 config는 그 안에 없음을 명시.
- `skills/github/SKILL.md`: Config/PAT 확인 안내에 `⚠️` 블록 추가 — `github_cli.py` 전용 캐시 `ls` 패턴과 config 경로를 분리, 캐시를 뒤지면 등록된 PAT를 다시 묻게 됨을 경고.
- `skills/report/SKILL.md`: PAT 확인 단계에 `⚠️` 블록 추가 — `report_cli.py` 탐색용 캐시 `ls` 패턴을 config 찾기에 쓰지 말 것.
- `skills/commit/SKILL.md`: 자동 승인 모드 판정의 config 읽기 줄에 인라인 보강 — 고정 경로 한 곳만 보고 `ls`·glob·플러그인 캐시 탐색 금지.
- `skills/changelog-deploy/SKILL.md`: `Read` 도구로 config를 읽는 절차 줄에 인라인 보강 — 고정 경로 한 곳만, 캐시(스크립트 전용)에 config 없음을 명시.
- `skills/ssh/SKILL.md`: Config 파일 안내 줄(`ssh` 섹션)에 인라인 보강 — 고정 경로를 Read tool로 바로 읽고 `ls`·glob·플러그인 캐시 탐색 금지.
- `skills/synology-expose/SKILL.md`: config 절차(§2~5) 안내에 `⚠️` 블록 추가 — 고정 경로 Read, `ls`·glob·플러그인 캐시 탐색 금지.

## 주요 구현 내용

- **단일 진실 원천 강화**: 모든 규칙의 출처는 `config-rules.md §3`의 격상된 경고 블록 하나다. 기존에도 §3에 "탐색 금지" 규칙이 있었으나 약하게 묻혀 에이전트가 건너뛰었기 때문에, `⚠️` 경고 블록으로 격상해 가시성을 높였다.
- **스킬별 인라인 이중 방어**: 7개 스킬은 각자의 진입 지점에서 참조 문서를 거치지 않고 시작될 수 있으므로, 각 SKILL.md의 config/PAT 확인 지점에 같은 취지의 짧은 방어 표기를 인라인으로 심었다. 각 스킬에는 자기 스킬의 실제 CLI 스크립트명(`issue_cli.py`·`github_cli.py`·`report_cli.py` 등)을 넣어 "이 캐시 ls는 스크립트 전용"이라는 점을 구체적으로 못 박았다.
- **두 경로의 명시적 분리**: 스크립트는 플러그인 캐시(`~/.claude/plugins/cache/...`)에서 `ls`로 찾고, config는 홈의 고정 경로(`{HOME}/.suh-template/config/config.json`)에서 Read로 읽는다는 원칙을 모든 표기에서 일관되게 반복해, 두 탐색 경로가 섞이지 않도록 했다.
- **스크립트 탐색 ls 로직 무변경**: 스킬들이 실제로 스크립트를 찾을 때 쓰는 `ls ~/.claude/plugins/cache/...` 실행 로직(코드 블록)은 한 줄도 수정하지 않았다. 이번 변경은 그 로직 주변에 설명/안내 텍스트만 추가한 것이다.

## 주의사항

- 이번 변경은 **문서/안내 텍스트만 추가**한 작업으로, 실행 로직(스크립트 탐색용 캐시 `ls` 패턴, CLI 호출 등)은 전혀 변경하지 않았다. 따라서 기존 스크립트 동작·CLI 호출 흐름에는 영향이 없다.
- config 경로는 표기상으로만 다룬다 — 실제 PAT 값 등 민감정보는 문서에 포함하지 않으며, 경로 표기만 노출한다.
- 인라인 방어 표기는 단일 진실 원천(`config-rules.md §3`)의 보조 수단이다. 규칙을 갱신할 때는 §3의 경고 블록을 기준으로 삼고, 스킬별 인라인 문구가 그와 어긋나지 않도록 함께 유지한다.
