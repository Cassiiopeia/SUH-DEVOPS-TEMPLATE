# deploy_branch 필드가 이름과 정반대 개념을 가리키던 문제 정리

## 개요

마법사 브랜치 질문(`deploy_branch`)이 이름과 실제 용도가 어긋나 사용자가 혼란을 겪던 문제를 정리했다. 이 필드는 코드상 "릴리스 PR의 head", 즉 개발한 걸 모아 기본 브랜치로 올리는 **개발 브랜치(develop)** 를 담는데, 이름과 마법사 라벨은 "배포 브랜치"였다. "배포"라 이름 붙여놓고 개발 브랜치 값(develop)을 기본으로 주니 사용자가 "배포인데 왜 develop?"으로 혼란에 빠졌다. 라벨·안내 문구를 실체(개발/릴리스 소스 브랜치)에 맞게 정정하고, 감지된 기본 브랜치를 동적으로 노출하도록 했다.

## 변경 사항

### 질문 라벨·안내 정정
- `src/core/options-ask.js`: "🌿 릴리스 배포 브랜치" 질문을 "개발한 코드를 모아서 배포로 올리는 '개발 브랜치'"로 정정. "배포 브랜치" 단어 제거. git으로 감지한 기본 브랜치를 함께 노출("감지된 기본(배포) 브랜치는 'main'입니다").
- 브랜치 생성 안내(`ensureDeployBranch`)도 "배포 브랜치" → "개발(릴리스 소스) 브랜치"로 정정. push 질문에 브랜치명 명시(#481 연계).

### 개념 문서화
- `CLAUDE.md`: 브랜치 3개념 표(기본/배포 브랜치·개발/릴리스 소스 브랜치·이슈 브랜치)와 필드 매핑을 추가. `deploy_branch`가 이름과 달리 개발 브랜치임을 명시하고, 사용자 노출 문구에서 "배포 브랜치"로 부르지 말 것을 규칙화.

### 테스트
- `test/branch-strategy.test.js`: 브랜치 생성 안내에 "개발" 표현 사용, push 질문에 브랜치명 명시(#481) 검증 보강.

## 주요 구현 내용

- **필드명 개명은 이번에 안 함**: `deploy_branch` → `release_source_branch` 개명은 version.yml 하위호환·스킬(changelog_cli) 소비처까지 파급이 커, 라벨·문구·문서 정합만 먼저 하고 필드 개명은 후속 과제로 남겼다.
