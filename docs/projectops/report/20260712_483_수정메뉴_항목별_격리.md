# 수정 메뉴에서 배포/Publish 방식만 골라도 옵션 6종이 전부 재질문되던 버그 수정

## 개요

분석 개요 카드에서 "수정하기 → 배포/Publish 방식"을 골랐을 때, 배포 방식 하나만 다시 묻는 게 아니라 deploy·publish·CodeRabbit·changelog·릴리스 브랜치·Secret 옵션이 전부 다시 나오던 문제를 해결했다. 원인은 수정 메뉴의 단일 항목이 `askAllOptionalWorkflows(forceAsk: true)`로 매핑되어 있었고, 이 함수가 6종 옵션을 한 덩어리로 묻기 때문이었다. 수정 메뉴를 항목별로 분해하고, 함수에 "어떤 축만 재질문할지" 스코프 개념을 도입해 한 항목 수정 = 그 항목만 재질문이 되도록 했다.

## 변경 사항

### 축 스코프 도입
- `src/core/options-ask.js`: `OPTION_AXES` 상수(deploy·publish·code-review·changelog·release-branch·secret)와 `scope` 인자 추가. `ask(axis)` 헬퍼가 "전역 forceAsk이고, scope가 없거나 그 축을 포함할 때만" 강제 질문하도록 각 옵션 가드를 교체. scope=null이면 전 축 재질문(초기 통합·기존 동작 보존).

### 수정 메뉴 분해
- `src/ui/prompts.js`: 기존 "배포/Publish 방식" 단일 항목을 6개 항목(배포 방식·라이브러리 배포·CodeRabbit·changelog·릴리스 소스 브랜치·Secret)으로 분해.
- `src/commands/interactive.js`: 선택한 항목을 `scope: [what]`으로 넘겨 그 축만 재질문. 나머지 옵션은 현재값 그대로 유지.

### 테스트
- `test/options-ask.test.js`: 축 격리 3종 — scope=[deploy]면 deploy만 재질문(나머지 유지), scope=[code-review]면 CodeRabbit만, scope=null이면 전 축 재질문(기존 동작).

## 주요 구현 내용

- **격리의 핵심은 함수 시그니처가 아니라 가드 조건**: 각 옵션의 `forceAsk || X === null` 가드를 `ask(axis) || X === null`로 바꾼 것만으로 스코프 밖 축은 저장값을 유지하고 질문을 건너뛴다.
- scope=null 경로를 남겨 초기 통합(전체 질문)과 CLI 비대화형 동작을 100% 보존.
