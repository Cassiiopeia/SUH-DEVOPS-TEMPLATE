📝 현재 문제점
---

- 댓글 기반 빌드 트리거의 동작이 프로젝트 타입별로 비대칭이다.
- Spring·Python의 PR-Preview는 `@suh-lab server build <브랜치명>` 형태로 빌드할 브랜치를 직접 지정할 수 있다(`custom_branch` 방식).
- 반면 Flutter 앱빌드 트리거(`@suh-lab build app` / `apk build` / `ios build`)는 댓글 본문을 빌드 타입(app/apk/ios) 판별에만 사용하고, 브랜치는 PR head 또는 이슈의 `Guide by SUH-LAB` 댓글에서 자동 추출한다.
- 따라서 `@suh-lab build app feature/xxx` 처럼 브랜치명을 적어도 뒤의 브랜치명이 무시되어, 원하는 특정 브랜치를 빌드할 수 없다.

🛠️ 해결 방안 / 제안 기능
---

- Flutter 앱빌드 댓글 트리거에 특정 브랜치명 인자를 지원한다.
- 명령어 문법:
  - `@suh-lab build app <브랜치명>` → 명시 브랜치로 Android + iOS 빌드
  - `@suh-lab apk build <브랜치명>` → 명시 브랜치로 Android만
  - `@suh-lab ios build <브랜치명>` → 명시 브랜치로 iOS만
  - 브랜치명을 생략하면 기존대로(PR head / 이슈 Guide 댓글) 동작.
- 브랜치 결정 우선순위: 명시 브랜치 > PR head > 이슈 Guide 댓글. (Spring/Python `custom_branch`와 동일한 책임 분리)
- 빌드 위치 추적(결과 댓글이 달리는 PR/이슈 번호, 빌드 번호의 소스 번호)은 명시 브랜치와 무관하게 댓글 컨텍스트를 그대로 유지한다.
- 관련 이슈 번호는 빌드 대상 브랜치명의 `#숫자`에서 추출한다.

⚙️ 작업 내용
---

- 대상 파일: `.github/workflows/project-types/flutter/PROJECT-FLUTTER-SUH-LAB-APP-BUILD-TRIGGER.yaml` (단일 파일)
- 빌드 타입 판별 스텝에서 명령어 키워드 뒤 토큰을 옵셔널 브랜치 인자로 정규식 캡처
  - `@suh-lab\s+build\s+app(?:\s+(\S+))?`
  - `@suh-lab\s+apk\s+build(?:\s+(\S+))?`
  - `@suh-lab\s+ios\s+build(?:\s+(\S+))?`
- 트리거 `if:` 조건(`contains` 기반)은 변경하지 않는다. 정규식은 판별 스텝 내부에서만 적용.
- 명시 브랜치도 기존 브랜치 존재 확인(getBranch) 스텝으로 동일하게 검증하고, 미존재 시 기존 에러 댓글 스텝을 재사용.
- 하위 빌드 워크플로우(`PROJECT-FLUTTER-ANDROID-TEST-APK`, `PROJECT-FLUTTER-IOS-TEST-TESTFLIGHT`)는 이미 `branch_name`을 `client_payload`로 받으므로 수정 불필요.
- 공통 워크플로우가 아니므로 `.github/workflows/` 루트 동기화는 불필요(타입별 워크플로우는 `project-types/flutter/`만 유지).
