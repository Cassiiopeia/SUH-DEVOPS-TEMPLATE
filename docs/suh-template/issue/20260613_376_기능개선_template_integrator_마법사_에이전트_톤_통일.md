📝 현재 문제점
---

`template_integrator.sh` / `template_integrator.ps1` 마법사의 사용자 대상 출력이 **톤이 혼재**되어 있어, 사용자가 각 단계에서 "지금 무엇을 왜 하는지" 직관적으로 따라오기 어렵다.

- **규모**: 사용자 대상 출력 호출이 `.sh` 한 파일에만 ~400개(`print_step` 46, `print_info` 143, `print_success` 43, `print_warning` 48, `print_to_user` 121, `print_question_header` 4), `.ps1`까지 합치면 ~800개 수준이다.
- **톤 혼재**: 일부는 이미 에이전트형 안내 톤으로 작성돼 있으나(예: `show_project_type_menu`의 `🔍 이 레포를 살펴봤습니다:`), 대부분은 기계적 로그 톤(`타입별 프로젝트 경로 확인 중...`)이라 한 마법사 안에서 말투가 들쭉날쭉하다.

**대표 사례 — 멀티타입 경로 설정 단계 (`resolve_project_paths` / `Resolve-ProjectPaths`)**

1. **도입부 없이 곧바로 항목으로 진입**: 멀티타입(spring, flutter, react, python)이 감지됐는데, "무엇이 감지됐고 이제 무엇을 차례로 할 것인지" 설명 없이 바로 `[1/4] spring: ...`으로 들이댄다. 사용자는 맥락을 잡기 전에 질문부터 받는다.
2. **경로 기준이 불명확**: `spring 경로를 'server'로 설정할까요?`에서 `server`가 *어디 기준의 어떤 경로*(레포 루트 기준 상대경로? 빌드 경로? 소스 경로?)인지 사용자가 알 수 없다. 실제 의미는 "레포 루트 기준 `server/` 폴더 = `build.gradle`이 있는 프로젝트 루트"이지만 그 기준이 화면에 드러나지 않는다.

🛠️ 해결 방안 / 제안 기능
---

마법사 전체의 사용자 대상 출력을 **"에이전트가 사용자에게 설명하며 안내하는" 일관된 톤**으로 전수조사하여 통일한다.

**1. 톤 가이드 원칙 수립 (작업의 기준선)**

먼저 "에이전트형 안내 톤"이 무엇인지 정의하는 가이드 원칙을 세운다. 이후 모든 치환과 신규 출력이 이 기준을 따른다.

- **맥락 먼저**: 단계를 시작할 때 "지금 무엇을, 왜, 몇 단계에 걸쳐 하는지" 먼저 한두 줄로 설명한 뒤 세부 질문/항목으로 진입한다.
- **감지 결과 요약**: 자동 감지(타입, 버전, 경로 등) 결과를 사람이 읽기 쉽게 요약해서 보여주고 나서 확인을 받는다.
- **기준 명시**: 경로나 값을 물을 때 그 값이 *무엇을 기준으로 한 어떤 값인지* 화면에 드러낸다(예: 전체 경로 `<레포루트>/server/build.gradle` 노출).
- **일관된 말투와 이모지, 들여쓰기**: 기존 `print_*` 헬퍼의 이모지/들여쓰기 컨벤션은 유지하되 말투를 통일한다.

**2. 전수조사 후 단계별 톤 통일**

`.sh`와 `.ps1` 두 파일을 **동일하게** 다듬는다(한쪽만 고치면 OS별 경험이 갈린다). 주요 대상 단계:

- 프로젝트 타입 자동 감지 / 선택 메뉴
- 버전 정보 자동 감지
- **타입별 경로 설정**(대표 사례 — 도입부 추가 + 경로 기준 명확화)
- 템플릿/워크플로우/스크립트 다운로드 단계
- version.yml, README 생성 단계
- IDE Skills, 플러그인 관리 단계

**3. 대표 사례 구체화 (경로 설정 단계)**

- 멀티타입 감지 시: 감지된 타입 목록을 표로 요약 → "이제 하나씩 차례대로 각 프로젝트의 루트 디렉터리를 설정하겠습니다" 선언 → 항목 진입.
- 각 항목: `'server'`만 던지지 않고 `<레포루트>/server/build.gradle` 전체 경로를 함께 노출해 "어디 기준의 어떤 경로"인지 명확히 한다.
- 직접 입력 프롬프트: "(`build.gradle`이 있는 폴더, 예: server, app — 루트면 Enter)"처럼 어떤 마커 파일이 있는 폴더인지 명시한다.

⚙️ 작업 내용
---

- 톤 가이드 원칙 문서화(작업 기준선)
- `template_integrator.sh` 사용자 대상 출력 전수조사 표 작성(단계 → 현재 톤 → 개선 톤)
- `.sh` 단계별 출력 톤 통일 적용
- `.ps1` 동일 적용(문구와 흐름 1:1 일치 확인)
- macOS 검증: `.sh`는 `expect`로 TTY 동작, `.ps1`은 Docker PowerShell 파서로 구문 + 함수 단위 입력 주입 동작 검증(CLAUDE.md 검증 절차 준수)
- 대표 사례(경로 설정 단계) 도입부와 경로 기준 명확화 우선 적용 후 전체로 확장
