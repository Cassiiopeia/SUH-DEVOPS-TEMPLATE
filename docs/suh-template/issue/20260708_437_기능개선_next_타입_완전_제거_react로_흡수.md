📝 현재 문제점
---

`next`가 `react`와 별도 프로젝트 타입으로 분리되어 있으나, **CI·배포 워크플로우가 사실상 동일**하다. 타입을 하나 더 유지하는 비용만 발생한다.

REACT vs NEXT 워크플로우 실측 대조:

| 항목 | REACT | NEXT | 차이 |
|------|-------|------|------|
| CI 빌드 | `npm ci` → `npm run build` | 동일 | 없음 |
| `.next/cache` 캐싱 | 있음 | 있음 | 없음 (react도 이미 next 캐싱) |
| 배포 | Docker 이미지 → SSH 서버 | 동일 | 없음 |
| 컨테이너명/prefix | `-front` / `front-container` | `-nextjs` / `nextjs-container` | 문자열만 |
| SSR 옵션 | 없음 | `NODE_ENV=production`, `--restart unless-stopped` | 사소 |

- 결정적 증거: `PROJECT-REACT-CICD`의 job 이름이 이미 `"React/Next.js 애플리케이션 빌드"`로 **둘을 같이 표기**하고 있다. 원저자도 동일한 것으로 취급했다.
- `PROJECT-REACT-CI`는 이미 `.next/cache` 캐싱을 수행한다 — react 워크플로우가 이미 next를 커버한다.
- next를 별도 타입으로 두면 감지 로직에서 "next를 react보다 먼저 체크"하는 순서 의존성, `version_manager` case 문, `.sh`/`.ps1`/`.js` 3중 분기를 계속 유지해야 한다. 95% 동일한 것을 5% 차이 때문에 이중 관리한다.

관련 파일:
- `.github/workflows/project-types/next/` (PROJECT-NEXT-CI, PROJECT-NEXT-CICD)
- `.github/workflows/project-types/react/` (흡수 대상)
- `src/core/detect.js` (`classifyPackageText`의 `"next"` 분기)
- `template_integrator.sh` / `.ps1` (`VALID_TYPES`, `classify_package_json`, `detect_project_types`)
- `.github/scripts/version_manager.sh` (case 문의 `next`)

🛠️ 해결 방안 / 제안 기능
---

**`next` 타입을 완전히 제거하고 `react`로 흡수한다. alias 하위호환은 두지 않는다.**

- 감지: `package.json`에 `react`가 있으면 `react`로 판정한다. `"next"` 우선 분기를 삭제한다.
- `VALID_TYPES`에서 `next` 제거.
- next 전용 워크플로우 폴더 삭제. 단 **next의 SSR용 옵션(`NODE_ENV=production`, `--restart unless-stopped`)을 통합된 react 배포 워크플로우에 흡수**하여 next 프로젝트도 그대로 동작하게 한다.
- `version_manager.sh` / `detect.js` / `.ps1`의 `next` 분기 제거.

**하위호환을 지원하지 않는다.**

- 기존 `project_type(s)`에 `next`가 설정된 프로젝트는 `react`로 수동 변경하도록 안내한다.
- **v4.0.0 breaking change**로 처리하고 `breaking-changes.json`에 등록한다 (severity: critical). 전환 절차: `version.yml`의 타입을 `next` → `react`로 변경 후 재통합.

> 데이터모델 정리([이중화 제거 이슈])와 함께 v4.0.0 릴리스로 묶는다.
