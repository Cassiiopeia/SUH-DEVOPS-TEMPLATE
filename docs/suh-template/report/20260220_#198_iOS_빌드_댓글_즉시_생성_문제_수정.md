### 📌 작업 개요

`@suh-lab app build` 커맨드 실행 시, iOS 빌드의 진행 상황 댓글이 즉시 달리지 않는 문제 수정.
Android는 `ubuntu-latest` 러너에서 바로 댓글이 생성되지만, iOS는 `macos-26` 러너 할당 대기 시간(수 분)으로 인해 댓글 생성이 지연됨.
추가로 Android/iOS 양쪽 워크플로우에서 준비(prepare) 실패 시 댓글이 "진행 중..." 상태로 영원히 남는 문제도 함께 해결.

### 🔍 문제 분석

**원인**: iOS 워크플로우(`PROJECT-FLUTTER-IOS-TEST-TESTFLIGHT.yaml`)의 `prepare-test-build` job이 `macos-26` 러너에서 실행되며, 이 job 안에서 진행 상황 댓글을 생성하고 있었음. macOS 러너는 할당까지 수 분이 걸리므로, 그 시간 동안 사용자에게 아무런 피드백이 없는 상태 발생.

**비교**: Android는 `ubuntu-latest`(즉시 할당)에서 댓글을 생성하므로 문제 없음.

### ✅ 구현 내용

#### 1. `notify-start` job 신규 추가 (iOS 워크플로우)
- **파일**: `.github/workflows/project-types/flutter/PROJECT-FLUTTER-IOS-TEST-TESTFLIGHT.yaml`
- **변경 내용**: `ubuntu-latest`에서 실행되는 `notify-start` job을 새로 추가하여 댓글을 즉시 생성
- **이유**: macOS 러너 할당 대기 없이 즉시 PR/이슈에 진행 상황 댓글을 달기 위함

#### 2. `prepare-test-build` job 조건 변경 (iOS 워크플로우)
- **파일**: `.github/workflows/project-types/flutter/PROJECT-FLUTTER-IOS-TEST-TESTFLIGHT.yaml`
- **변경 내용**: `needs: notify-start` 의존성 추가 및 `branch_valid` output 기반 조건 분기
- **이유**: 브랜치 유효성과 댓글 생성을 분리하여, 댓글 API 실패 시에도 빌드가 중단되지 않도록 구현

#### 3. 하위 job들의 skip chain 방지 (iOS 워크플로우)
- **파일**: `.github/workflows/project-types/flutter/PROJECT-FLUTTER-IOS-TEST-TESTFLIGHT.yaml`
- **변경 내용**: `build-ios-test`와 `deploy-testflight-test`에 `if: !cancelled() && needs.*.result == 'success'` 조건 추가
- **이유**: `workflow_dispatch`로 수동 실행 시 `notify-start`가 skip되어 하위 job들이 전부 skip되는 문제 방지

#### 4. 준비 실패 댓글 업데이트 추가 (iOS + Android 워크플로우)
- **파일**: `.github/workflows/project-types/flutter/PROJECT-FLUTTER-IOS-TEST-TESTFLIGHT.yaml`
- **파일**: `.github/workflows/project-types/flutter/PROJECT-FLUTTER-ANDROID-TEST-APK.yaml`
- **변경 내용**: `prepare-test-build` job에 `if: failure()` 조건의 "준비 실패" 댓글 업데이트 스텝 추가
- **이유**: prepare 단계 실패 시 댓글이 "⏳ 진행 중..." 상태로 영원히 남는 문제 해결

### 🔧 주요 변경사항 상세

#### iOS 워크플로우 구조 변경

**변경 전** (job 3개):
```
prepare-test-build (macos-26) → build-ios-test → deploy-testflight-test
     ↑ 브랜치 확인 + 댓글 생성 + 환경 설정 모두 여기서 처리
```

**변경 후** (job 4개):
```
notify-start (ubuntu-latest) → prepare-test-build (macos-26) → build-ios-test → deploy-testflight-test
     ↑ 브랜치 확인 + 댓글 생성           ↑ 환경 설정만
```

#### notify-start job 주요 설계
- `repository_dispatch` 이벤트일 때만 실행 (`workflow_dispatch`에서는 skip)
- 브랜치 존재 여부를 먼저 확인하고, 실패 시 에러 댓글 작성 후 `core.setFailed`
- 브랜치 확인 성공 시 `branch_valid: 'true'` output 설정
- 댓글 생성 스텝에 `continue-on-error: true` 적용 → 댓글 API 실패 시에도 빌드 진행 가능

#### prepare-test-build 조건 분기
```yaml
if: |
  always() &&
  (needs.notify-start.result == 'skipped' || needs.notify-start.outputs.branch_valid == 'true')
```
- `skipped`: `workflow_dispatch`로 수동 실행 시 → 그대로 진행
- `branch_valid == 'true'`: `repository_dispatch`에서 브랜치 확인 성공 시 → 진행
- 그 외 (브랜치 없음 등): 실행하지 않음

#### comment_id 참조 경로 변경
모든 comment 업데이트 스텝에서 참조 경로 변경:
- **변경 전**: `steps.progress.outputs.comment_id` / `steps.progress.outputs.start_time`
- **변경 후**: `needs.notify-start.outputs.comment_id` / `needs.notify-start.outputs.start_time`

#### Android 워크플로우 준비 실패 댓글
`prepare-test-build` job 내에 `if: failure()` 조건의 스텝 추가. 실패 시 댓글을 "❌ 준비 실패" 상태로 업데이트하여 사용자에게 즉시 피드백 제공.

**특이사항**:
- Android는 이미 `ubuntu-latest`에서 실행되므로 `notify-start` 분리가 불필요
- Android의 댓글 참조는 기존 `steps.progress.outputs.*` 방식 그대로 유지

### 🧪 테스트 및 검증

- `repository_dispatch` 시나리오: `notify-start`가 즉시 실행되어 댓글 생성 → `prepare-test-build`가 `branch_valid == 'true'`로 진행 → 이후 job들 정상 실행
- `workflow_dispatch` 시나리오: `notify-start`가 skip → `prepare-test-build`가 `skipped` 조건으로 진행 → `build-ios-test`와 `deploy-testflight-test`가 `!cancelled() && needs.*.result == 'success'` 조건으로 정상 실행
- 브랜치 미존재 시나리오: `notify-start`에서 에러 댓글 작성 후 `core.setFailed` → `branch_valid` 미설정 → `prepare-test-build` 미실행
- 댓글 API 실패 시나리오: `continue-on-error: true`로 `notify-start` 자체는 성공 → `branch_valid == 'true'`로 빌드 진행 → 댓글 업데이트 스텝들은 `comment_id != ''` 조건으로 자동 skip
- prepare 실패 시나리오: "준비 실패" 댓글 업데이트 → 사용자에게 즉시 피드백

### 📌 참고사항
- `PROJECT-FLUTTER-SUH-LAB-APP-BUILD-TRIGGER.yaml`은 변경 불필요 (이미 브랜치 확인 후 에러 댓글 처리가 정상적으로 구현되어 있음)
- 소요 시간 계산에는 macOS 러너 할당 대기 시간이 포함됨 (notify-start의 start_time 기준)
- 이는 사용자 관점에서 "요청 후 실제 걸린 시간"을 보여주는 것이므로 의도된 동작
