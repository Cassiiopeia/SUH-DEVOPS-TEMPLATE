### 📌 작업 개요
`synology-expose` skill 제작 과정에서 사용자 피드백을 반복 적용하며 좋은 skill을 만들기 위한 테크닉과 원칙을 정리한 리포트

---

### 🎯 목표
시놀로지 NAS에 웹 서비스를 외부 도메인으로 노출하는 반복 작업을 skill로 템플릿화하여, 누구나 사용할 수 있는 범용 가이드 skill 제작

---

### ✅ 피드백 기반 개선 내역

#### 1. 초기 설정 질문 방식: "한꺼번에 묻지 말고 하나씩"

**문제**: 처음에 도메인, DDNS, DNS 제공자, 레코드 타입, 이메일, 저장 위치를 한 번에 물어봄. 사용자 입장에서 압도적이고, 전문 용어를 모르면 답할 수 없음

**개선**: `superpowers:brainstorming` skill의 "한 번에 하나만 묻는다" 원칙을 접목
- 한 메시지에 질문 하나만
- 선택지 제공 (번호 선택)
- 응답에 따라 후속 질문 결정
- 질문 개수를 미리 고정하지 않음

**적용한 brainstorming 원칙**:
```
- 한 번에 하나만 묻는다
- 선택지를 제공한다
- 응답에 따라 후속 질문을 결정한다
- 전문 용어를 설명한다
```

**핵심 교훈**: 질문을 고정 N개로 한정하면 안 된다. 사용자 응답에 따라 추가 질문이 필요할 수도, 불필요할 수도 있다. brainstorming skill처럼 대화형으로 유연하게 진행해야 함

---

#### 2. 불필요한 질문 제거: "자동 판단할 수 있으면 묻지 마라"

**문제**: DDNS 주소를 이미 받았는데 DNS 레코드 타입(CNAME/A)을 또 물어봄

**개선**: DDNS 주소(문자열) → CNAME, 고정 IP(숫자) → A 레코드로 자동 판단. 별도 질문 삭제

**핵심 교훈**: 사용자가 이미 준 정보에서 추론 가능한 건 묻지 않는다. 질문 하나를 줄이는 것이 UX를 크게 개선한다

---

#### 3. 전문 용어 설명: "모르는 사람도 있다고 가정하라"

**문제**: "Let's Encrypt 이메일"이라고만 물어보면, "Let's Encrypt에 회원가입해야 하나?" 같은 혼란 발생 가능

**개선**: 
- "별도 회원가입은 필요 없고, 인증서 만료 알림을 받을 이메일만 입력하면 됩니다"
- "시놀로지가 자동 갱신하므로 알림은 거의 오지 않습니다"

**핵심 교훈**: skill은 다양한 수준의 사용자가 쓸 수 있다. 전문 용어를 쓸 때는 반드시 한 줄 설명을 덧붙인다

---

#### 4. 구체적인 UI 안내: "어디에 뭐가 있는지 정확히 알려줘라"

**문제**: "시놀로지 DDNS 주소를 알려주세요"만으로는 어디서 확인하는지 모름

**개선**: DSM 스크린샷 기반으로 구체적 경로와 컬럼명까지 안내
```
확인 방법:
  시놀로지 DSM > 제어판 > 외부 액세스 > DDNS 탭
  - "호스트 이름": DDNS 주소 → CNAME 레코드에 사용
  - "외부 주소": 현재 공인 IP → A 레코드에 사용 (고정 IP인 경우)
```

**핵심 교훈**: UI 기반 가이드 skill은 실제 화면의 필드명, 버튼명, 경로를 정확히 써야 한다. 스크린샷을 기반으로 작성하면 정확도가 높아진다

---

#### 5. 하드코딩 제거: "개인 정보를 skill에 박아넣지 마라"

**문제**: 예시에 실제 도메인(`suhsaechan.kr`), DDNS 주소(`suh-project.synology.me`) 등이 하드코딩됨

**개선**: 모든 예시를 `{도메인}`, `{ddnsAddress}` 등 플레이스홀더로 변경. 실제 값은 설정 파일에서 읽어서 실행 시점에 채움

**핵심 교훈**: 
- 공용 템플릿 skill에는 개인 정보를 절대 하드코딩하지 않는다
- 보안 측면에서도 DDNS 주소, 이메일 등이 노출되면 안 된다
- 예시가 필요하면 `example.com`, `my-nas.synology.me` 같은 범용 예시를 사용한다

---

#### 6. 실사용 경험 기반 보정: "문서가 아니라 경험을 믿어라"

**문제**: "Cloudflare Proxied 선택 시 인증서 발급에 추가 설정 필요"라고 안내했지만, 실제로는 문제 없었음

**개선**: 사용자의 실제 경험을 반영
- Proxied도 인증서 발급 문제 없음 (추가 설정 불필요 문구 삭제)
- 서브도메인에 `.`이 포함된 경우만 Proxied 사용 불가 (실제 Cloudflare DNS 목록 스크린샷에서 확인)

**핵심 교훈**: 일반적으로 알려진 정보보다 사용자의 실제 운영 경험이 더 정확할 수 있다. skill에 잘못된 경고를 넣으면 오히려 혼란을 준다

---

#### 7. 서비스 정보 자동 추론: "줄 수 있는 건 다 받고, 부족한 것만 물어봐라"

**문제**: 서비스명, 서브도메인, 포트, 웹소켓 여부를 하나하나 물어보는 방식

**개선**: 사용자가 한번에 여러 정보를 줄 수 있으므로 자동 추론 우선
- "mongo-express를 올려야해" + docker-compose → 서비스명, 포트, 웹소켓 여부 모두 자동 추출
- "grafana 3000번 포트 올려줘" → 서비스명, 포트 자동 추출, 서브도메인만 확인
- 특정 소스(docker-compose 등)에 의존적으로 작성하지 않음

**핵심 교훈**: 초기 설정과 서비스 정보 수집은 성격이 다르다. 초기 설정은 하나씩 물어봐야 하지만, 서비스 정보는 사용자가 자유롭게 주는 대로 받고 부족한 것만 추가 질문한다

### 🏗️ 좋은 Skill을 만들기 위한 원칙 (정리)

#### 질문 설계
| 원칙 | 설명 |
|------|------|
| **한 번에 하나씩** | 여러 질문을 한 메시지에 넣지 않는다 |
| **선택지 제공** | 번호 선택이나 예시를 보여준다 |
| **자동 판단 우선** | 추론 가능한 건 묻지 않는다 |
| **유연한 흐름** | 질문 개수를 고정하지 않고 응답에 따라 결정한다 |
| **전문 용어 설명** | 모르는 사람도 있다고 가정한다 |

#### 콘텐츠 설계
| 원칙 | 설명 |
|------|------|
| **하드코딩 금지** | 개인 정보, 실제 도메인 등을 skill에 박아넣지 않는다 |
| **설정 파일 분리** | 사용자별로 다른 값은 설정 파일로 분리한다 |
| **실경험 반영** | 문서보다 실제 운영 경험을 우선한다 |
| **UI 경로 구체화** | 버튼명, 필드명, 탭 이름까지 정확히 안내한다 |

#### 구조 설계
| 원칙 | 설명 |
|------|------|
| **brainstorming 패턴 접목** | 대화형 질문, 응답 기반 분기를 skill에 적용 |
| **초기 설정 vs 실행 분리** | 첫 사용 시 설정 생성, 이후에는 설정 읽어서 바로 실행 |
| **조건부 섹션** | 웹소켓 등 선택적 기능은 조건부로 안내 |
| **다중 도메인 지원** | 배열로 관리하여 확장성 확보 |

### 🔗 superpowers:brainstorming에서 가져온 핵심 패턴

| brainstorming 원칙 | synology-expose에 적용한 방식 |
|---------------------|-------------------------------|
| 한 번에 하나만 묻는다 | 초기 설정 질문을 단계별로 분리 |
| 선택지 제공 | DNS only/Proxied, 저장 위치 등 번호 선택 |
| 응답에 따라 후속 질문 결정 | Cloudflare → Proxy status 질문, A 레코드 → IP 질문 |
| 전문 용어 설명 | Let's Encrypt, CNAME/A 레코드 설명 |
| 모르면 기본값 추천 | "잘 모르겠다" → 기본값 추천 + 이유 설명 |

이 패턴은 절차 안내형 skill뿐 아니라, 사용자 입력이 필요한 모든 skill에 범용적으로 적용 가능하다.
